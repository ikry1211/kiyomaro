# YouTube長尺動画 自動編集パイプライン

> **ステータス:** 調査・設計段階
> **最終更新:** 2026-04-15
> **関連:** [動画制作自動化パイプライン（リール版）](/plans/video-automation) — 短尺動画の自動化戦略 | [YouTubeチャンネル運営戦略](/plans/youtube-channel-strategy) — コンテンツ戦略・制作ワークフロー

---

## 1. 概要

### 1.1. 目的

YouTube にアップロードする長尺動画（10分〜）の編集工程を自動化し、制作コストと工数を大幅に削減する。

### 1.2. チャンネル戦略: 2チャンネル体制

**パイプラインは1つ、テンプレートとアップロード先だけ分ける**構成で、2つのチャンネルを運営する。

| | CH1: ブラウザゲーム開発 | CH2: 3Dプリンタ ものづくり |
|:---|:---|:---|
| **テーマ** | AIと協業してブラウザゲームを作る | AIと議論しながらコードで3D設計 → 3Dプリントで実体化 |
| **制作手段** | コード（JS/TS/Canvas/WebGL） | コード（OpenSCAD/CadQuery等） |
| **素材の性質** | スクリーンレコーディング | コード + 3Dプレビュー + 実写 |
| **撮影の手間** | ✅ 低い（OBS録画で済む） | △ 3Dプリント・組立は実写 |
| **視聴者体験** | **URL1つで即プレイ可能** ← 強い | STLファイル公開で再現可能 |
| **開始優先度** | ✅ **先行**（撮影不要で始めやすい） | 横展開（パイプライン確立後） |

> **チャンネルを分ける理由**: YouTubeのアルゴリズムはチャンネルのテーマ一貫性を重視する。テーマが混在すると「何のチャンネルか」が不明になり、レコメンドされにくくなる。

#### 共通の特徴

- どちらも「**AIと議論しながらコードで作る**」プロセスが中心
- 制作物がコードなので **Git管理・diff表示・LLM解析** が容易
- パイプラインのコア部分は共通で、テーマ固有のテンプレートだけ差し替え

### 1.3. リール版との違い

| | リール版（既存） | YouTube長尺版（本ドキュメント） |
|:---|:---|:---|
| **尺** | 15〜60秒 | 10〜30分 |
| **用途** | クライアント受託 | 自社チャンネル |
| **素材** | スマホ撮影（縦型） | 画面録画 + 実写（横型） |
| **音声** | BGM + テロップ中心 | ナレーション（TTS or 肉声） |
| **編集の複雑さ** | テンプレート流し込み | カット編集 + 構成 + 合成 |
| **ツール構成** | Remotion 中心 | **FFmpeg + Remotion + Gemini解析 併用** |

## 2. 技術構成

### 2.1. ツール役割分担

```

```

| 工程 | FFmpeg | Remotion |
|:---|:---|:---|
| 無音カット・トリミング | ✅ | |
| 音声抽出・正規化 | ✅ | |
| 倍速処理 | ✅ | |
| テロップ・モーション生成 | | ✅ |
| イントロ/アウトロ生成 | | ✅ |
| パーツ一覧・スペック表示 | | ✅ |
| 全素材の最終結合 | ✅ | |
| エンコード・書き出し | ✅ | |

### 2.2. 全体パイプライン

```
📝 台本作成（人間 or LLM補助）
  ↓
📹 素材録画（OBS画面録画 / カメラ撮影）
  ↓
🤖 Gemini でシーン解析 ← ★ LLMによる「意味理解」
  ├── シーン分割・分類
  ├── ハイライト検出
  ├── 不要カット特定
  └── 構成提案（JSON出力）
  ↓
👤 人間レビュー（解析結果の確認・修正）
  ↓
✂️ 自動カット編集（FFmpeg）← 解析JSONに基づいて実行
  ├── 無音区間カット
  ├── フィラー除去（「えーと」「あの」等）
  └── 作業シーンの倍速化
  ↓
🗣️ TTS ナレーション生成（Gemini 2.5 Flash）
  ↓
📝 字幕生成（Whisper）
  ↓
🎨 テロップ・図解生成（Remotion）
  ↓
🎵 BGM・SE合成（FFmpeg）
  ↓
📊 サムネイル生成（テンプレート）
  ↓
⬆️ YouTube自動アップロード（YouTube Data API v3）
```

### 2.3. LLM動画解析（Gemini）

FFmpegだけでは「無音検出」「映像変化量」等の信号処理しかできない。「何が映っているか」「どのシーンを残すか」の**意味理解**にはマルチモーダルLLMが不可欠。

| 解析タスク | FFmpegだけ | Gemini必要 |
|:---|:---|:---|
| 無音区間の検出 | ✅ | |
| 映像の変化量 | ✅ | |
| **「何をしている場面か」の理解** | ❌ | ✅ |
| **失敗テイクの判定** | ❌ | ✅ |
| **ハイライト検出** | ❌ | ✅ |
| **倍速すべき区間の判断** | △ | ✅ |
| **不要カットの特定** | ❌ | ✅ |
| **構成の提案** | ❌ | ✅ |

> **なぜ Gemini か**: 動画ファイルをネイティブに入力できるLLMは現状 Gemini が最も成熟。フレーム抽出等の前処理が不要で、最大1時間程度の動画を直接解析可能。

#### 解析プロンプトのイメージ

Geminiに素材動画を投入し、以下のJSONを出力させる：

```json
{
  "scenes": [
    {"start": "00:00:00", "end": "00:02:30", "summary": "コードの解説", "importance": 4},
    {"start": "00:02:30", "end": "00:05:00", "summary": "コーディング作業", "importance": 2, "speedup": "4x"},
    {"start": "00:05:00", "end": "00:05:15", "summary": "カメラ調整（不要）", "importance": 0, "action": "cut"}
  ],
  "highlights": [
    {"time": "00:18:30", "description": "ゲームが初めて動いた瞬間"}
  ],
  "suggested_structure": "完成デモ → 設計議論 → 実装 → テストプレイ"
}
```

→ このJSONを元にFFmpegのカット・倍速・結合コマンドを**自動生成**する。

#### 解析コスト

| モデル | 30分素材の解析 | 月4本 |
|:---|:---|:---|
| Gemini 2.5 Flash | $0.05〜0.20/回 | $0.20〜0.80 |
| Gemini 2.5 Pro（高精度） | $0.50〜2.00/回 | $2.00〜8.00 |

### 2.4. 各工程の自動化レベル

| 工程 | 自動化 | 手法 | 備考 |
|:---|:---|:---|:---|
| 素材管理・振り分け | ◎ | Python/Shell スクリプト | 撮影日・カメラ別に自動整理 |
| **シーン解析・構成提案** | **○** | **Gemini マルチモーダル** | **JSON出力 → 人間レビュー** |
| 無音カット | ◎ | FFmpeg 音声解析 | |
| フィラー除去 | ○ | Whisper → 「えーと」等を検出 | |
| 作業シーン倍速化 | ○ | **Gemini解析 + FFmpeg** | Gemini併用で精度向上 |
| 字幕・テロップ | ◎ | Whisper → SRT → Remotion | |
| 強調テロップ | ○ | キーワード検出で自動挿入 | |
| BGM配置 | ○ | 場面プリセットで自動配置 | |
| SE（効果音） | △ | テロップ出現時のみ自動化可 | |
| 音量自動調整 | ◎ | FFmpeg loudnorm/compressor | |
| サムネイル | △〜○ | テンプレート差し込み生成 | |
| 構成の最終判断 | △ | Geminiが提案 → **人間が最終判断** | |
| 書き出し | ◎ | FFmpeg バッチ処理 | |
| YouTube アップロード | ◎ | YouTube Data API v3 | |
| タイトル・説明文 | ○ | LLM で自動生成 | |

## 3. TTS（テキスト読み上げ）

### 3.1. 推奨構成

| 用途 | サービス | 理由 |
|:---|:---|:---|
| **メインナレーション** | Gemini 2.5 Flash（Native Audio） | プロンプトでトーン指示可能。コスパ良好 |
| **技術用語の読み上げ** | Google Cloud TTS（Neural2） | 安定性・正確性重視 |
| **キャラクター利用** | VOICEVOX（ずんだもん等） | エンタメ寄りにする場合 |

### 3.2. TTS活用パターン

| パターン | 内容 | 適性 |
|:---|:---|:---|
| **A. 全体TTS** | 台本 → TTS → 音声ナレーション。顔出しなしチャンネル向き | ○ |
| **B. 補足のみTTS**（推奨） | 本人の声 + 定型部分だけTTS | ◎ |
| **C. 多言語展開** | 日本語撮影 → Whisper → LLM翻訳 → TTS（英語版） | ○ |

### 3.3. TTS比較

| サービス | 自然さ | 日本語品質 | 感情制御 | コスト |
|:---|:---|:---|:---|:---|

## 4. インフラ戦略

### 4.1. フェーズ別アーキテクチャ

#### Phase 1: ローカル完結（まず動かす）

```
Mac（ローカル）
├── FFmpeg: カット・結合・エンコード
├── Whisper: 音声認識（CPU版 or MLX版）
├── Remotion: テロップ生成
├── TTS: Gemini API 呼び出し
└── YouTube API: アップロード
```

- **コスト**: ほぼ0円（API費用のみ）
- **メリット**: 素材転送不要、セットアップ最も簡単
- **デメリット**: レンダリング中はPCが重い

#### Phase 2: クラウド移行（ワークフロー安定後）

**選択肢A: Cloud Run Jobs（コスト重視）**

```
Mac（ローカル）              Cloud Run Jobs（GCP）
├── 素材管理                 ├── FFmpeg エンコード
├── 台本作成         ─GCS─→  ├── Remotion レンダリング
├── 人間レビュー     ←GCS─  ├── Whisper 処理
└── 最終確認                 └── 最終書き出し
```

**選択肢B: Trigger.dev（DX重視）**

```
Mac（ローカル）              Trigger.dev（Cloud）
├── 素材管理                 ├── パイプラインオーケストレーション
├── 台本作成         ─S3─→   ├── FFmpeg / Remotion 処理
├── 人間レビュー     ←通知─  ├── wait.forToken()でレビュー待ち
└── 最終確認                 └── チェックポイント・リトライ
```

### 4.2. インフラコスト比較

#### コンピュート単価（同等スペック）

| スペック | Trigger.dev | Cloud Run | 差 |
|:---|:---|:---|:---|
| 1 vCPU / 秒 | $0.0000675 | $0.0000240 | Cloud Run 64%安い |
| 2 vCPU / 秒 | $0.0001700 | $0.0000480 | Cloud Run 72%安い |
| 4 vCPU / 秒 | $0.0003400 | $0.0000960 | Cloud Run 72%安い |

#### 月間コスト試算（10分動画）

| 構成 | 月4本 | 月8本 | 月20本 |
|:---|:---|:---|:---|
| **ローカルのみ** | ¥500（電気代） | ¥500 | ¥500 |
| **Cloud Run Jobs** | ¥80（ほぼ無料枠内） | ¥160 | ¥400 |
| **Trigger.dev (Cloud)** | ¥230 | ¥460 | ¥1,160 |
| **VPS (Hetzner AX41)** | ¥5,600（固定） | ¥5,600 | ¥5,600 |

> ※ 上記はコンピュート費用のみ。API費用（TTS、Whisper、LLM）は別途 ¥1,000〜3,000/月 程度

#### Cloud Run 無料枠

- 毎月 180,000 vCPU-秒（≒ 50時間分）
- 毎月 360,000 GiB-秒
- 月4本程度なら **無料枠内に収まる可能性が高い**

#### Trigger.dev の付加価値

| 機能 | Trigger.dev | Cloud Run |
|:---|:---|:---|
| ワークフローオーケストレーション | ✅ 組み込み | ❌ 別途 Workflows 等が必要 |
| 人間レビュー待ち（wait.forToken） | ✅ 待機中は課金ゼロ | ❌ 自前実装必要 |
| リトライ / エラーハンドリング | ✅ 宣言的 | △ Job単位のリトライのみ |
| チェックポイント | ✅ 自動 | ❌ なし |
| ダッシュボード | ✅ リッチ | △ Cloud Logging |
| 開発体験 | ✅ TypeScript first | △ Docker 必須 |

### 4.3. 推奨方針

1. **Phase 1はローカルで**パイプラインのプロトタイプを構築（コストほぼ0）
2. ワークフローが安定したら **Cloud Run Jobs に移行**（コスト最小）
3. パイプラインが複雑化したり、チーム運用が必要になったら **Trigger.dev を検討**

## 7. チャンネル別の追加自動化

### 7.1. ブラウザゲーム開発チャンネル

| 自動化 | 手法 |
|:---|:---|
| **Git diff → コード差分アニメーション** | Remotionでdiffをシンタックスハイライト付きで表示 |
| **コード解説の自動生成** | LLMにコードを渡して解説テロップを自動生成 |
| **ビルド結果の自動キャプチャ** | ゲーム実行画面を自動録画 |
| **進捗サマリ** | Gitログから「今回の実装内容」を自動生成 |
| **プレイアブルリンク** | 動画説明欄にデプロイ済みURLを自動挿入 |

### 7.2. 3Dプリンタ ものづくりチャンネル

| 自動化 | 手法 |
|:---|:---|
| **Git diff → 設計変更の可視化** | コミットごとにモデルをレンダリング → 連番で並べる |
| **3Dモデル回転アニメーション** | OpenSCADコマンドラインレンダリング or Three.js |
| **AI議論ダイジェスト** | チャットログをLLMで要約 → テロップ化 |
| **BOM（部品表）自動生成** | コードからパーツリスト・3Dプリント時間を抽出 |
| **STLファイル自動配布** | 動画公開と同時にGitHub/Thingiverseに自動公開 |

## 8. リスク

| リスク | 深刻度 | 対策 |
|:---|:---|:---|
| FFmpegエンコードが想定より重い | 🟡 | ローカルで実測してからクラウド移行判断 |
| Remotion長尺レンダリングのメモリ問題 | 🟡 | セグメント分割レンダリングで回避 |
| TTS音声の不自然さ | 🟡 | Gemini Flash のトーン指示で調整 / 肉声との併用 |
| Geminiのシーン解析精度 | 🟡 | プロンプト調整とFew-shot例の追加で改善 |
| 素材転送のボトルネック（クラウド移行時） | 🟡 | 回線速度を事前確認。GCS へのアップロードを非同期化 |
