# SFA 通話フィードバック機能 — 要件仮説

> **背景:** 2026年3月の通話品質分析（手動実施）で、成果を分ける要因が「会話の構造」にあることが判明。この知見をSFAに組み込み、フィードバックサイクルを自動化する際の要件仮説をまとめる。
>
> **ステータス:** 仮説段階。4月の手動フィードバック効果を検証後、開発判断。

---

## 1. データ基盤

### 1.1 通話データの蓄積

| 項目 | 現状 | SFA化 |
|:---|:---|:---|
| データソース | Amazon Connect → Slack Bot投稿 | Amazon Connect → **DB直接蓄積** |
| 保存期間 | Slackの履歴に依存 | 永続化（月次・四半期比較可能に） |
| 取得可能範囲 | MCP経由で直近〜1000件程度 | 全件。日付・メンバー・結果でクエリ可能 |

### 1.2 蓄積すべきデータ項目

**通話メタデータ:**
- 日時、担当者、架電先、通話時間、結果（アポ/不在/断り等）

**会話構造データ（自動算出）:**
- ターン数、AGENT発話文字数（平均）、質問数、質問種別（定型/深掘り）
- 断りシーン数、断り後の継続有無
- CUSTOMER/AGENT感情推移

**会話内容データ（LLM分析）:**
- アプローチの型（過去接点引用/簡潔型/説明先行型）
- ヒアリングの有無と深度
- 提案内容の分類（デモ誘導/機能+価格/コスト削減等）
- 断り対応パターン（受容→質問/資料送付/即終了等）
- クロージングの型（その場確定/布石/なし）

## 2. 定量ダッシュボード（週次自動更新）

### 2.1 メンバー別KPI

今回の分析で成果との相関が確認された指標を自動集計：

| KPI | 成果上位の水準 | 閾値アラート |
|:---|:---|:---|
| 断り後の継続率 | 80〜83% | **60%未満で警告** |
| 1通話あたり質問数 | 3.1〜3.8回 | **2.0未満で警告** |
| 1発話の平均文字数 | 33〜47文字 | **70文字超で警告** |
| CUSTOMER感情悪化率 | 31%（日高） | **65%超で警告** |
| クロージング質問率 | 12%（日高） | 参考値として表示 |

### 2.2 トレンド表示

- 週次の推移グラフ（改善提案前後の変化を可視化）
- メンバー間比較チャート

---

## 3. 質的フィードバック（LLM自動生成）

### 3.1 分析フレームワーク（5軸）

今回の手動分析で有効性が確認された5軸をそのままフレームワークに：

| 軸 | 自動化の方針 |
|:---|:---|
| **A. アプローチ** | LLMで冒頭3ターンを分類（過去接点引用/簡潔型/一方的説明型） |
| **B. ヒアリング** | 質問の抽出と分類。説明前に質問があるかの順序判定 |
| **C. 提案** | サービス名・価格・機能・事例の言及有無を自動検出 |
| **D. 断り対応** | 断りシーン前後の会話パターンを分類（受容→質問/繰り返し/即終了） |
| **E. クロージング** | 日程提案/名前取得/曖昧終了の分類 |

### 3.2 個別フィードバックの自動生成

- 週次でメンバーごとのフィードバックテキストを生成
- 強み + 改善ポイント + 具体的なアクション提案の構造
- 前週との比較を含める（「先週より質問数が1.2回増えました」等）

## 4. 配信機能

| 配信先 | 内容 |
|:---|:---|
| **Slack DM** | 週次の要約（KPI + 改善ポイント1つ） |
| **SFA画面** | 詳細フィードバック（5軸の分析 + トレンド） |
| **マネージャー向け** | チーム全体サマリー + メンバー間比較 |

---

## 5. フォローアップ管理

今回の分析で明らかになった「布石型クロージング」（野島）の効果を最大化する機能：

- 通話中に取得した**担当者名・連絡先・次回架電予定日**を自動抽出
- 再架電タスクを自動生成（期日・優先度付き）
- 過去の通話履歴を次回架電時に表示（「前回5月にラクコを紹介済み」等）

## 6. 開発優先度（仮説）

| フェーズ | 機能 | 理由 |
|:---|:---|:---|
| **Phase 1** | データ蓄積基盤（1.1, 1.2） | 全ての分析の前提。Slack依存からの脱却 |
| **Phase 2** | 定量ダッシュボード（2.1, 2.2） | 手動集計の自動化。週次で改善を追跡 |
| **Phase 3** | フォローアップ管理（5） | 布石型クロージングの回収漏れ防止 |
| **Phase 4** | 質的フィードバック自動生成（3） | LLM活用。手動フィードバックの知見を基に |
| **Phase 5** | 配信自動化（4） | Slack DM + SFA画面への配信 |

## 7. 検証ポイント（4月に確認）

手動フィードバックの効果を確認し、SFA化の判断材料とする：

- [ ] 福田さんの発話長が85文字→50文字以下に改善されたか
- [ ] 福田さんのヒアリング質問数が増加したか
- [ ] 藤本さんの感情悪化率が73%→改善されたか
- [ ] 鍵谷さんの断り後継続率が57%→改善されたか
- [ ] フィードバックの配信手段（PDF手動 vs Slack vs Notion）の運用負荷

> **判断基準:** 手動フィードバックで1名以上のKPIに改善が見られれば、自動化する価値がある。
