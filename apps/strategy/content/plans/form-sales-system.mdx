# フォーム営業の仕組み設計

> **ステータス:** 企画段階
> **最終更新:** 2026-03-28
> **親ドキュメント:** [マーケティング事業計画 §3.2, §7 #8](/business/marketing)
> **設計思想:** 極限まで自動化 → やむを得ず人力が残る部分だけB型事業所に格安委託

---

## 1. 目的と設計思想

### 1.1. なぜフォーム営業か

マーケティング事業Layer 2（AI活用型広告運用 ¥25万/月）の**主軸獲得チャネル**。追加コストほぼゼロ。

| チャネル | 予算 | 役割 |
|:---|:---|:---|
| **フォーム営業**（自動化 + B型委託） | ≒ ¥0 | 主軸。広告出稿中の企業を狙い撃ち |
| Web広告（粗利の25%） | ¥10万〜 | CPA/CVR検証 + 面でのリーチ |

### 1.2. 自動化ファーストの原則

**「自動化できるところは全てAI/スクリプトで自動化し、どうしても人力が必要な部分だけB型事業所に格安で委託する」**

```
自動化の優先順位:

  🤖 完全自動化を目指す
    ├── ターゲットリスト作成（広告ライブラリ巡回・企業情報収集）
    ├── 企業情報の分析・スコアリング
    ├── 営業文面のパーソナライズ生成
    ├── フォームURL の自動検出
    ├── 送信結果の記録・リード管理
    └── 返信検知・通知

  👤 人力が残る可能性が高い部分 → B型事業所に委託
    ├── フォーム入力・送信（CAPTCHAやフォーム構造のバラつき）
    └── フォームが見つからない場合の手動検索
```

> フォーム送信自体もヘッドレスブラウザ等で自動化を試みるが、**CAPTCHA・reCAPTCHA・フォーム構造の多様性**により完全自動化は困難な可能性がある。その場合のみB型事業所の出番。

### 1.3. SOCTにおけるシナジー

仕組みは**SOCT全体の営業インフラ**として設計。ターゲットと文面を差し替えるだけで横展開可能。

| 事業 | ターゲット | 訴求 |
|:---|:---|:---|
| **マーケティング事業** | 広告出稿中の企業 | 代理店手数料¥50万→¥25万に削減 |
| **はなえみダイレクト** | 法定雇用率対象企業（40名+） | 障害者雇用マッチングPFの無料利用 |
| **Rakuco** | テレアポの補完 | 福祉SaaSの案内 |

> 最初はマーケティング事業で構築・検証し、パターン確立後に他事業へ。

## 3. Step 1: ターゲットリスト自動生成 🤖

### 3.1. ターゲット定義（マーケティング事業）

| 条件 | 理由 |
|:---|:---|
| 広告出稿中の企業 | 広告予算あり = 切り替え提案が成立 |
| 手数料¥50万+ | ¥25万への切替で年間¥300万+削減の訴求 |
| 中規模企業（EC・サービス業中心） | 大企業は代理店固定、小規模は手数料が低い |

### 3.2. 自動リスト生成の手段

| 手段 | 自動化可否 | 精度 | 概要 |
|:---|:---|:---|:---|
| **Meta広告ライブラリAPI** | 🟢 API公開 | 🟡 | 出稿企業・広告内容・期間を自動取得 |
| **Google広告透明性センター** | 🟡 スクレイピング | 🟡 | 出稿主の検索が可能 |
| **リスティング広告の検索結果** | 🟢 自動化容易 | 🟢 | 特定キーワード検索→広告表示企業を自動収集 |
| **SimilarWeb API** | 🟡 有料 | 🟢 | 広告出稿量・チャネル分析 |
| **BuiltWith等** | 🟡 有料 | 🟡 | 広告タグ検出で出稿中か推定 |
| **業界リスト・ランキング** | 🟢 スクレイピング | 🟢 | EC/D2Cランキング等。広告出稿がほぼ確実 |

### 3.3. AIスコアリング

リストを取得したら、AIで各企業をスコアリングし優先順位をつける。

| スコア要素 | 重み | データソース |
|:---|:---|:---|
| 推定広告予算規模 | 高 | 広告ライブラリの出稿頻度・期間 |
| 業種フィット | 高 | EC・D2C・サービス業に加点 |
| 企業規模（従業員数） | 中 | 企業サイト・DB |
| 既存代理店使用の推定 | 中 | LP品質・広告運用の特徴から推定 |

### 3.4. リストのデータ項目

| 項目 | 自動取得 | 備考 |
|:---|:---|:---|
| 企業名 | 🤖 | — |
| 業種 | 🤖 | AI分類 |
| WebサイトURL | 🤖 | — |
| フォームURL | 🤖（後述） | Step 3で自動検出 |
| 推定広告予算規模 | 🤖 | スコアリング |
| 広告出稿チャネル | 🤖 | Meta/Google等 |
| スコア | 🤖 | 優先順位 |

## 4. Step 2: 文面パーソナライズ自動生成 🤖

### 4.1. 生成プロセス

```
インプット（全て自動収集済み）:
  - 企業名、WebサイトURL、業種、広告出稿状況

AI処理:
  1. 企業サイトをクロール → 事業内容・商材・ターゲット顧客を抽出
  2. 訴求テンプレート × 企業情報 → パーソナライズ文面を生成
  3. 品質チェック（文字数・禁止ワード・トーン）

アウトプット:
  - 件名 + 本文（300〜500文字）
  → スプレッドシートに自動書き出し
```

## 5. Step 3: フォーム検出 + 送信 🤖→👤

**ここが自動化の最大の技術的課題。**

### 5.1. 自動化で解決できる部分

| 処理 | 技術 | 実現可能性 |
|:---|:---|:---|
| **フォームURL自動検出** | クローラーが企業サイトを巡回し `お問い合わせ` ページを検出 | 🟢 高い |
| **フォーム構造の解析** | HTMLパースで入力フィールド（名前・メール・本文等）を特定 | 🟡 中程度 |
| **自動入力・送信** | Puppeteer/Playwright等のヘッドレスブラウザで操作 | 🟡 フォーム依存 |

### 5.2. 自動化が困難な部分（→ B型事業所に委託）

| 障壁 | 発生頻度（推定） | 理由 |
|:---|:---|:---|
| **CAPTCHA / reCAPTCHA** | 🔴 30〜50% | 自動送信の最大の壁。API回避はグレー |
| **非標準フォーム構造** | 🟡 10〜20% | JavaScript動的生成、iframe埋め込み等 |
| **フォームが見つからない** | 🟡 5〜10% | メールのみ・電話のみの企業 |
| **必須項目の判断** | 🟡 10% | プルダウン・ラジオボタン等の選択が必要 |

> **推定: 全体の40〜60%は自動送信可能。残り40〜60%がB型事業所の手動送信の対象。**
> 自動化率は実装・検証を進めるにつれ改善できる可能性あり。

### 5.3. ハイブリッド運用フロー

```
🤖 自動パイプライン:
  1. フォームURL検出
  2. フォーム構造解析
  3. Playwrightでフォームに自動入力
     ├── ✅ CAPTCHAなし → 自動送信 → 完了
     └── ❌ CAPTCHAあり / 入力失敗 → 「確認キュー」へ

👤 B型事業所（確認キュー処理）:
  4. Webアプリ上でAI入力済みのフォームを確認
  5. CAPTCHAがあれば対応
  6. 内容を確認して送信ボタンを押す
  7. 完了チェック → 次のタスクが自動表示
```

> **核心UX: AIが入力、人が確認・送信。** B型ワーカーの作業は「確認してボタンを押す」だけに簡素化できる。技術方式は§5.4で比較検討。

### 5.4. 送信UXの技術選択肢

「AI入力済みのフォームをB型ワーカーに確認・送信させる」を実現する方式の比較。v0で検証して決定する。

#### 方式A: Chrome拡張

B型ワーカーのローカルChromeに拡張をインストール。拡張がフォームを開いて自動入力し、ワーカーが確認・送信。

- ✅ VPS不要。コスト¥0
- ✅ ローカルブラウザなのでレスポンスが速い
- ✅ DOM直接操作でフォーム入力精度が高い
- ⚠️ B型のPCに拡張インストールが必要（限定公開URLで配布可能）
- ⚠️ 各端末での動作を一元管理しにくい

#### 方式B: Playwright + リモートブラウザ（自前）

サーバー上のPlaywrightがフォームを開いて自動入力。noVNCまたはBrowserless OSSでブラウザ画面をWebアプリに配信。

- ✅ B型は拡張インストール不要。Webアプリを開くだけ
- ✅ サーバーで一元管理・監視
- ✅ B型のPC性能に依存しない
- ⚠️ ConoHa VPS（¥2K/月〜）が必要
- ⚠️ noVNC or Browserless OSSの構築が必要

#### 方式C: クラウドブラウザサービス

Browserbase等のAPIでクラウドブラウザを起動し、埋め込みURLをWebアプリに表示。

- ✅ インフラ構築不要。API呼び出しだけ
- ✅ スケールが容易
- ⚠️ 従量課金（月1,000セッションで$10〜30 ≒ ¥1,500〜4,500）
- ⚠️ 外部依存

#### 比較まとめ

| | Chrome拡張 | Playwright自前 | クラウドブラウザ |
|:---|:---|:---|:---|
| **月額コスト** | **¥0** | ¥2K〜 | ¥1.5〜4.5K |
| **構築難易度** | ★★ 低 | ★★★ 中 | **★ 最低** |
| **B型の導入** | 拡張インストール | **ブラウザだけ** | **ブラウザだけ** |
| **一元管理** | ⚠️ 難しい | **✅ 容易** | **✅ 容易** |
| **スケール** | 端末追加 | VPS増強 | **自動** |
| **推奨フェーズ** | **v0（最速検証）** | v1〜（本格運用） | v0（プロトタイプ） |

> **v0の方針:** Chrome拡張かクラウドブラウザで最速検証 → v1以降の本格運用で方式を確定。

## 7. B型事業所委託の設計

### 7.1. 委託範囲

**自動化パイプラインで処理できなかった「人力キュー」のみを委託。** リスト作成・文面生成・リード管理は全て自動化済みの前提。

| B型事業所が行うこと | B型事業所が行わないこと |
|:---|:---|
| AI入力済みフォームの確認・送信 | ターゲット選定 |
| CAPTCHAの対応 | 文面の作成・編集 |
| 送信完了のチェック | 商談対応 |
| フォームが見つからない場合の報告 | フォームへの入力（AIが実施） |

### 7.2. Rakucoとのシナジー

| メリット | 詳細 |
|:---|:---|
| **圧倒的に安い** | B型工賃は¥500/h程度。通常の業務委託（¥1,500〜3,000/h）の1/3〜1/6 |
| **Rakuco顧客基盤** | 既存の関係性から打診しやすい |
| **PC作業の提供** | B型事業所にとって良質なPC作業の確保は価値が高い |
| **スケーラブル** | 人力キューの量に応じて委託量を調整可能 |

### 7.3. 品質管理

| 論点 | 対策 |
|:---|:---|
| 送信ミス | AIが入力済み。B型は確認と送信のみ。自由入力を排除 |
| 進捗管理 | Webアプリ上のタスク一覧で自動追跡 |
| 個人情報 | リストに個人名を含まない。企業名・URL・文面のみ |
| 送信ペース | 1日50件程度の上限設定。スパム判定回避 |

## 9. ファネル数値・コスト構造・ROI

### 9.1. フォーム営業ファネル

> 業界ベンチマーク + marketing.mdx §6 のユニットエコノミクスを基に試算。v0で実測し修正する。

```
送信  →  返信  →  商談  →  トライアル  →  有料転換
100%     0.5〜2%    30〜50%    50%            30%
```

| ステップ | 転換率 | 想定根拠 |
|:---|---:|:---|
| **送信 → 返信** | **0.5〜2%** | 定型文の一斉送信は0.1〜0.5%。パーソナライズ有りで0.5〜1.5%が現実的。2%は上位ケース |
| **返信 → 商談設定** | **30〜50%** | 返信＝一定の興味あり。吉岡さんの商談力で上振れ期待 |
| **商談 → トライアル申込** | **50%** | 3ヶ月無料 + コスト削減の明確な訴求。ハードルが低い |
| **トライアル → 有料転換** | **30%** | marketing.mdx §4.3 の保守的採用値（目標50%、業界15〜25%） |

> **返信率は低い。だからこそ自動化で送信件数をスケールさせ、量で獲得数を確保する。** これがフォーム営業×自動化の戦略の本質。

### 9.2. フェーズ別シミュレーション

#### 前提

| パラメータ | 値 | ソース |
|:---|---:|:---|
| 有料月額 | ¥250,000 | marketing.mdx §4.2 |
| SOCT粗利率 | 40% | 吉岡さん報酬60%後 |
| 月次粗利/社 | ¥100,000 | ¥25万 × 40% |
| LTV（解約率20%） | ¥5,400,000 | marketing.mdx §6.1 |
| LTV（解約率30%） | ¥3,400,000 | marketing.mdx §6.1 |

#### ファネル試算（返信率1%・中央値ケース）

**自動化の核心は送信件数のスケール。** 返信率が低くても、コストほぼゼロで大量送信すれば数で補える。

| Phase | 月間送信 | 返信（1%） | 商談（40%） | トライアル（50%） | 有料転換（30%） | 月次売上増 |
|:---|---:|---:|---:|---:|---:|---:|
| **v1**（検証） | 1,000 | 10.0 | 4.0 | 2.0 | **0.6社/月** | **¥150K** |
| **v2**（スケール） | 1,000+ | 10.0+ | 4.0+ | 2.0+ | **0.6社+/月** | **¥150K+** |

**12ヶ月累計:**

| Phase | 12ヶ月送信 | 有料獲得 | 月次売上積上 | 年間粗利増 |
|:---|---:|---:|---:|---:|
| **v1（1,000件/月）** | **12,000** | **7.2社** | **¥1,800K** | **¥8,640K** |

> **v1（月1,000件）なら12ヶ月で7.2社を獲得。** コスト増は月数万円のみ。送信件数を上げるほど1社あたりの獲得コストは下がる。

#### 感度分析（返信率 × 送信件数）

| 返信率 | v3（500件/月）12ヶ月 | v4（1,000件/月）12ヶ月 |
|:---|---:|---:|
| **0.5%**（悲観） | 1.8社 | 3.6社 |
| **1%**（中央） | 3.6社 | **7.2社** |
| **2%**（楽観） | 7.2社 | **14.4社** |

> 返信率0.5%でもv4なら12ヶ月で3.6社。**量で返信率の低さを補える。**

### 9.4. CAC（フォーム営業経由）

| スケール | 月間コスト | 12ヶ月コスト | 有料獲得（返信率1%） | **CAC** |
|:---|---:|---:|---:|---:|
| v2（200件） | ¥16K | ¥192K | 1.4社 | **¥137K** |
| **v3（500件）** | **¥38K** | **¥456K** | **3.6社** | **¥127K** |
| **v4（1,000件）** | **¥74K** | **¥888K** | **7.2社** | **¥123K** |

> 送信件数が増えるほどCACは下がる（自動化固定費の薄まり）。**v4ではCAC ¥123Kで年7社獲得。** Web広告のCAC（¥300〜500K）と比べて依然として**1/2〜1/4。**
> ⚠️ 初期パイプライン構築の開発工数は含まない（CEO内製の前提）。

### 9.5. LTV/CAC と投資判断

> marketing.mdx §6 のLTV試算と突き合わせ。v3（500件/月）ベース。

| 解約率 | LTV（粗利ベース） | CAC（v3） | **LTV/CAC** | 判定 |
|:---|---:|---:|---:|:---|
| 20% | ¥5,400K | ¥127K | **43倍** | 🟢 圧倒的 |
| 30% | ¥3,400K | ¥127K | **27倍** | 🟢 非常に健全 |
| 40% | ¥2,400K | ¥127K | **19倍** | 🟢 健全 |

> 最悪ケースでもLTV/CAC 19倍。B型委託コストを含めても**SaaS目安3倍を大幅に上回る。**

### 9.6. Web広告との比較まとめ

| 指標 | フォーム営業（v3: 500件） | フォーム営業（v4: 1,000件） | Web広告 |
|:---|---:|---:|---:|
| **月間コスト** | ¥38K | ¥74K | ¥100K |
| **月間有料獲得** | 0.3社 | **0.6社** | 0.25社 |
| **12ヶ月累計獲得** | 3.6社 | **7.2社** | 3社 |
| **CAC** | ¥127K | **¥123K** | ¥300〜500K |
| **LTV/CAC（解約30%）** | 27倍 | 28倍 | 7〜11倍 |

> **結論:** B型委託コスト込みでも、**フォーム営業のCACはWeb広告の1/2〜1/4**。v4（月1,000件）ならWeb広告と同等以下のコストで**倍以上の有料顧客を獲得**。両チャネル合計で**月0.85社 ≒ 年10社**。Layer 2単体で年間売上¥2,500K（¥25万 × 10社）に到達し、目標が見えてくる。

## 10. 実行計画

| Phase | 内容 | 自動化範囲 | 月間送信 |
|:---|:---|:---|---:|
| **v0: 自動化検証** | リスト自動生成+AI文面生成+フォーム自動送信を構築。CEOが自分で回して「どこに人が必要か」を見極める | リスト+文面+送信トライ 🤖 | — |
| **v1: B型巻き込み+検証** | 人力部分をB型に委託。1,000社に送信しファネルを検証 | 全工程 🤖+👤 | 1,000 |
| **v2: スケール** | 送信件数を維持・拡大。ファネル改善・文面最適化 | 全工程 🤖+👤 | 1,000+ |
| **横展開** | はなえみダイレクト等へパラメータ差し替え | 共通基盤 🤖 | 事業別 |

### v0で見極めること

| 確認事項 | 理由 |
|:---|:---|
| **CAPTCHAの発生率** | 自動送信の成功率がわからないとB型の作業量が見積もれない |
| **非標準フォームの割合** | 自動入力できないフォームのパターンを把握 |
| **フォームが見つからない企業の割合** | メールのみ・電話のみの企業がどの程度あるか |
| **B型に渡す作業の粒度** | 「コピペで送信」だけか、それ以上の判断が必要か |

### v1で検証すべきKPI

| KPI | 目標 | 検証方法 |
|:---|:---|:---|
| ターゲットリストの母数 | 月1,000件以上の候補 | Meta広告ライブラリ等からの自動取得件数 |
| AI文面の品質 | 違和感なし | CEO/吉岡さんレビュー |
| フォーム送信→返信率 | 0.5〜2% | 1,000社送信して計測（1%なら10件返信。統計的に有意） |
| 返信→商談設定率 | 30%+ | 返信のうち商談に進む割合 |
| B型委託の運用品質 | 送信ミス率 5%以下 | スプレッドシートで追跡 |

### スケールの上限制約

| 制約 | 影響 | 対策 |
|:---|:---|:---|
| **ターゲットリストの枯渇** | 広告出稿中の中規模企業には限りがある | 業種拡大・はなえみダイレクト等への横展開 |
| **スパム判定** | 同一ドメインから大量送信するとブラック化 | 送信元の分散・1日あたり上限設定 |
| **B型事業所のキャパシティ** | 人力キューが増えすぎると処理が追いつかない | 複数事業所への分散委託 |
| **吉岡さんの商談キャパ** | 返信が増えても商談対応に限界 | v3以降でアシスタント検討 |
