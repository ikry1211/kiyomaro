# 運営分析レポート自動生成ツール

> **ステータス:** 設計段階（別プロジェクトで実装予定）
> **最終更新:** 2026-04-22
> **関連:** [コンサルティング事業計画 §4](../business/consulting) / [ヒアリングシート](../guides/hearing-sheet)

---

## 1. 概要

就労継続支援A型/B型事業所の運営状況を「InBody（体組成計）の診断レポート」のようなビジュアルで可視化し、2回目商談で画面共有しながら提案に使うツール。

### 位置づけ

```
フェーズ0: 事前調査 → ワムネットデータ取得
フェーズ1: 初回商談 → ヒアリング（データ収集）
フェーズ2: レポート作成 ← 🔥 このツールが担う
フェーズ3: 2回目商談 → 画面共有でレポート提示
```

### デザインコンセプト

InBodyの診断レポートをモデルとした、**1ページで完結するビジュアルレポート**。

| InBodyの要素 | コンサルレポートへの応用 |
|:---|:---|
| 総合スコア（大きな数字） | 事業所の「運営健全度スコア」（100点満点） |
| レーダーチャート | 加算・減算・人員・売上・IT・ガバナンスの6軸 |
| 色分けバー（緑/黄/赤） | 各項目の達成度を信号色で表示 |
| 基準値との比較 | 同規模事業所の平均スコアとの比較 |

## 2. アーキテクチャ

### 2.1. データパイプライン（3段階）

```
Stage 1: 構造化抽出（LLM）
  ヒアリング議事録（非構造テキスト）
    → JSONに変換（ヒアリングシートの項目に沿って抽出）

Stage 2: テンプレートレンダリング（コード）
  ワムネットJSON + ヒアリングJSON
    → レポートテンプレートに流し込み
    → スコアリングロジックで各項目を自動採点

Stage 3: 分析コメント生成（LLM）
  各セクションごとに1〜2文の「所見」を生成
    → テンプレートの該当箇所に挿入
```

> **ポイント:** フォーマットの統一はLLMに任せず、**Stage 2のテンプレートエンジン（コード）が担保**する。LLMは「データ抽出」と「所見コメント生成」のみ。

### 2.2. 技術スタック

| 層 | 技術 | 理由 |
|:---|:---|:---|
| フロントエンド | **Next.js** | SSRでPDF化にも対応可能 |
| チャート | **Recharts** | React親和性が高い。描画結果をそのままPDF化可能 |
| PDF生成 | **html2canvas + jsPDF** | ブラウザ内で完結。サーバーにデータが残らない |
| スコアリング | **JSONスキーマ + TypeScript** | 全事業所で同じ基準 |
| データソース | **Notion API**（既存ワークスペース） | 事業所DB・ヒアリング結果をNotionで管理 |
| ホスティング | **Vercel** | Next.jsネイティブ。東京リージョン（`hnd1`） |

> **DB不要設計:** Notionが実質的なデータベースとして機能するため、SupabaseやVercel Postgres等の追加DBは不要。
> 100社スケールでもNotion APIの制限内（3req/sec）で十分運用可能。

### 2.3. インフラ構成

```
Notion（既存WS）                 Vercel（Next.js）
┌──────────────────┐           ┌──────────────────────┐
│ 📋 事業所DB      │──read────→│ /api/report          │
│ 📋 ヒアリング結果DB│           │   Notion APIで取得    │
└──────────────────┘           │   ↓ スコアリング      │
                               │ /report/[id]         │
                               │   Recharts描画        │
                               │   ↓                  │
                               │   html2canvas + jsPDF │
                               │   ↓                  │
                               │   📄 PDFダウンロード  │
                               │   → ローカル保存      │
                               └──────────────────────┘
                               DB: なし
                               Region: hnd1（東京）
```

**セキュリティ:**
- PDF生成はブラウザ内で完結 → サーバーにデータが残らない
- Notion APIの認証はサーバー側（Internal Integration Token）
- 生成されたPDFはローカルPCに保存 → 情報漏洩リスク最小

### 2.4. 将来の拡張（必要になったら）

| 拡張 | トリガー | 追加技術 |
|:---|:---|:---|
| LLM所見生成 | 手動コメントが負担になったら | Gemini API追加 |
| Notionへの自動転記 | 契約後の転記が手間になったら | Notion API（write） |
| ワムネット自動取得 | 事前調査の手動入力が負担になったら | Vercel Cron + スクレイピング |
| レポート蓄積・比較 | 前回との比較が必要になったら | Supabase追加 |

## 3. レポート構成

### 3.1. レイアウト（1ページ）

```
┌─────────────────────────────────────────────────┐
│  運営分析レポート: 〇〇事業所（B型）              │
│  作成日: 2026/04/22                              │
├────────────────────┬────────────────────────────┤
│                    │                            │
│   総合スコア       │    レーダーチャート（6軸）   │
│   ┌───────┐       │                            │
│   │  62   │       │        ★加算               │
│   │ /100  │       │    IT●     ●減算           │
│   └───────┘       │      ●   ●                │
│   🟡 要改善       │   ガバ●  ●売上             │
│                    │       人員                 │
├────────────────────┴────────────────────────────┤

│  減算リスク  ████░░░░░░ 35/100  🔴 要対応       │
│  売上・稼働  ██████░░░░ 62/100  🟡 改善余地     │
│  人材・定着  █████░░░░░ 55/100  🟡 改善余地     │
│  ガバナンス  ███░░░░░░░ 28/100  🔴 要対応       │
│  IT・DX     ██░░░░░░░░ 20/100  🔴 要対応       │
├──────────────────────────────────────────────────┤
│  重点改善ポイント（TOP 3）                        │
│  1. BCP未策定 → 報酬返還リスクあり               │
│  2. 福祉専門職員配置加算 → 有資格者あり未申請     │
│  3. 議事録管理が紙 → 情報透明化の余地大          │
└──────────────────────────────────────────────────┘
```

### 3.2. スコアリング（6軸 × 配点）

| 軸 | 配点 | 主な評価項目 |
|:---|:---|:---|
| **加算取得** | 20点 | 取得可能な加算のうち何%を取得しているか |
| **減算リスク** | 20点 | 3大減算 + 人員配置基準の対応状況 |
| **売上・稼働** | 20点 | 充足率、出席率、営業活動 |
| **人材・定着** | 15点 | 配置余裕、離職率、採用チャネル |
| **ガバナンス** | 15点 | 会議体制、報告体制、苦情把握 |
| **IT・DX** | 10点 | 記録ソフト、コミュニケーション、自動化 |

> **配点の根拠:** 加算・減算は直接的な収益インパクトが大きいため高配点。IT・DXは改善余地は大きいが収益への直結度は相対的に低い。
> ⚠️ **配点の最終決定は北澤のレビューが必要。**

### 3.3. 色分け基準

| スコア | 色 | ラベル |
|:---|:---|:---|
| 80〜100 | 🟢 | 良好 |
| 50〜79 | 🟡 | 改善余地あり |
| 0〜49 | 🔴 | 要対応 |

## 4. JSONスキーマ（概要）

> 完全なスキーマは実装時に定義。ここでは主要フィールドのみ記載。

```typescript
interface ReportData {
  // 基本情報（ワムネット由来）
  事業所名: string;
  種別: "A型" | "B型" | "A型B型併設";
  所在地: string;
  定員: number;
  現利用者数: number;

  // 加算（ワムネット + ヒアリング）
  加算: {
    処遇改善: { 区分: string; 上位移行検討: boolean };
    福祉専門職員配置: { 取得: boolean; 有資格者あり: boolean };
    就労移行支援体制: { 取得: boolean; 移行実績あり: boolean };
    // ... A型/B型専用加算
  };

  // 減算リスク（ヒアリング）
  減算: {
    BCP_感染症: { 策定済: boolean; 更新済: boolean };
    BCP_災害: { 策定済: boolean };
    情報公表: { 報告済: boolean; 最新: boolean };
    虐待防止: { 委員会: boolean; 研修: boolean; 担当者: boolean };
    人員配置: { 突合チェック実施: boolean };
  };

  // 売上・稼働（ヒアリング）
  稼働率: { 値: number; 把握方法: string };
  集客: { チャネル: string[]; 課題: string };

  // 人材（ヒアリング）
  人材: { 配置余裕: boolean; 離職率認識: string; 定着施策: string[] };

  // ガバナンス（ヒアリング）
  ガバナンス: { 定例会議: boolean; 議事録方法: string; 苦情把握: boolean };

  // IT（ヒアリング）
  IT: { 記録ソフト: string; 連絡手段: string; 自動化関心: boolean };
}
```

## 5. 運用ルール

| ルール | 詳細 |
|:---|:---|
| **商談時はデータを渡さない** | 画面共有で見せるのみ。PDFダウンロードは無効化 |
| **契約後にNotionで共有** | レポートの内容をNotionクライアントページに転記 |
| **スコアは絶対評価** | 他社比較は「同規模平均」との比較のみ。個別事業所名は出さない |
| **所見は断定しない** | 「〜の可能性があります」「確認が必要です」等の表現 |

## 6. 実装ロードマップ

### MVP（最小構成）

> **目標:** Notionに入力済みのデータから、InBodyスタイルのビジュアルPDFを1クリックで生成できる状態。

| ステップ | 内容 | 優先度 |
|:---|:---|:---|
| **Step 1** | NotionにヒアリングDB（テンプレート）を作成 | 🔴 高 |
| **Step 2** | JSONスキーマ定義 + スコアリングロジック | 🔴 高 |
| **Step 3** | レポートUIプロトタイプ（Recharts + モックデータ） | 🔴 高 |
| **Step 4** | Notion API連携（DBからデータ取得） | 🔴 高 |
| **Step 5** | PDF生成（html2canvas + jsPDF） | 🔴 高 |
