# レインズ自動検索SaaS — JV検討

> **ステータス:** 検討中（初期評価）
> **最終更新:** 2026-03-29

## 1. 案件概要

不動産仲介会社の担当者向けに、レインズ（REINS）の物件検索を自動化し、新着物件の速報通知を行うSaaSを共同事業（JV）として開発・提供する。

### 1.1. ターゲット

- 毎日「新着物件がないか」をレインズで手動検索している**仲介担当者**
- 検索条件の入力 → 検索実行 → URL取得 → LINE送信を繰り返すルーチン作業に疲弊している層

### 1.2. バリュープロポジション

| 提供価値 | 説明 |
|:---|:---|
| **速報性** | 他社より1分でも早く新着物件を顧客に提案できる＝成約率直結 |
| **ルーチン解放** | 毎朝のゴミ作業（条件入力→検索→コピペ→LINE送信）が消える |

### 1.3. 競合分析

既に複数のChrome拡張・RPAツールが存在する**レッドオーシャン寄り**の市場。ただし有料で確立されたプレイヤーは少なく、多くは無料βまたは機能限定。

#### 周辺領域のSaaS（間接競合）

機能的な直接の競合ではないが、「追客・営業支援」という名目で仲介会社の予算（月額数万円〜）を奪い合う存在。これらとの連携、あるいはこれらがカバーしていない「最新物件の自動取得」が我々の立ち位置になる。

| サービス名 | カテゴリ | 特徴 | 料金感 |
|:---|:---|:---|:---|
| **いえらぶCLOUD** | 総合型業務支援 | ポータル連動、CRM、HP作成まで全て網羅。不動産SaaSの最大手 | 要問合せ（数万円〜） |
| **PropoCloud** | 追客/自動提案 | 顧客の希望条件に合った物件を自動提案。売買メイン | 要問合せ |
| **KASIKA** | MA（マーケ） | 顧客のWeb上の行動履歴を可視化し、見込み客を自動育成 | 要問合せ |
| **Facilo** | クラウド接客 | 顧客とのLINEやメールでのやり取りを一元管理し、独自マイページで物件提案 | 要問合せ |

#### 競合から読み取れるポイント

1. **iimonが事実上の業界標準**: 3,500店舗・継続率99.2%。レインズ連携ツールの直接競合市場は既に大手がいる
2. **価格帯が二極化**: iimon（総額機能・月¥30,000級）vs Miru for REINS（単機能拡張・月¥3,000級）。**参入するならどちらのレイヤーで戦うかの選択が必要**
3. **無料が多い**: フォレスト（全宅連提携）・Hakoma（β）が無料で提供。**無料ツールと差別化できないと課金が難しい**
4. **間接競合の存在**: いえらぶやPropoCloud等、高機能・高額な追客SaaSを導入済みの会社に対し、別軸で入り込む余地がある
5. **全宅連提携のフォレストが無料レイヤーの壁**: 業界団体のお墨付きで無料。低価格帯の参入障壁として機能

#### 差別化についての現状認識

既存ツールはすべて「ブラウザ上でのレインズ検索効率化」に集中している。差別化の方向性として以下が考えられるが、**いずれも技術的実現性・市場ニーズの両面で未検証**。PoC前の最優先検討事項。

| 差別化案 | 技術的実現性 | 課題 |
|:---|:---|:---|
| 公開情報（SUUMO等）の監視による補助通知 | ◎ 実装可能 | SUUMOの情報はレインズより遅い（不動産会社が手動掲載するため）。**「それレインズでもう見た」と言われるリスクが高い** |
| 顧客ごとの条件マッチング＋自動提案 | ○ 実装可能 | CRM/追客領域に寄るためPropoCloud等と競合する |
| AI物件レポート生成 | △ | Miru for REINSが既に実装。後追いになる |
| レインズUIの徹底的な改善・高速化 | ◎ 実装可能 | 差別化にはなるが、フォレスト（無料）の壁がある |

> ⚠️ **核心的な課題**: 後発で参入する以上、iimonと同等価格帯で「iimonにない明確な武器」が必要。だがレインズの技術的制約（§2参照）により、**サーバーサイドでの24時間自動巡回・通知は推奨アプローチでは不可能**。Chrome拡張ベースでどこまで差別化できるかがこの事業の成否を分ける。

#### 実際の競合環境

ただし、市場規模を踏まえると**iimonと直接ぶつかる可能性は低い**。

| 指標 | 数値 |
|:---|:---|
| REINS会員 | **約130,000事業所** |
| iimon導入数 | **3,500店舗（全体の約2.7%）** |
| **残り未導入** | **約126,500事業所（97%以上）** |
| 本サービス初年度目標 | **30社** |

市場の97%はまだ有料ツールを導入していない。初年度30社の獲得にiimonの既存顧客を奪う必要はなく、**「まだ何も使っていない店舗」へのアプローチで十分**。

したがって実質的な競合は：
- ❌ iimon → 既存顧客の奪い合いにはほぼならない
- ⭕ **「手動でやっている現状」**（何も導入していない状態）が最大の競合
- ⭕ **フォレスト（無料）** →「無料で十分」と言われないかが本当の壁

> **結局は営業力の勝負**。プロダクトの差別化よりも、「まだ何も使っていない97%の店舗」にリーチできるかどうかが事業の成否を決める。羽富側の営業力・不動産業界のネットワークがこのJVの最重要資産。

### 1.5. 収益モデル（仮説）

## 2. レインズの技術的制約

APIが存在せず、外部からのアクセスを排除する仕様になっている。

| 制約 | 内容 |
|:---|:---|
| **IP遮断** | AWS等のデータセンターIPからのアクセスは即BAN |
| **1ID 1セッション** | 同時ログインすると一方が強制ログアウト |
| **画像認証（CAPTCHA）** | 定期的に出現するボット対策 |
| **規約** | データの二次利用・転売は厳禁 |

## 3. 技術アプローチの選定

### 3.1. 推奨: Chrome拡張方式

競合もこの方式で運用しており、法的リスクと開発コストのバランスが最も優れる。

**仕組み:**
- ユーザー自身のブラウザ上で動作する拡張機能
- 検索条件の保存、新着物件のハイライト、ワンクリックLINE送信等のUI強化
- ユーザーの正規セッション・IP・ブラウザをそのまま使用

**法的リスク:**

| リスク | 深刻度 | 説明 |
|:---|:---|:---|
| 規約グレーゾーン | 🟡 中 | ブラウザ補助ツールだが、自動操作は規約的にはグレー |
| 不正アクセス禁止法 | 🟢 低 | CAPTCHAは人間が対応。アクセス制御の回避に該当しない |
| 顧客リスク | 🟢 低 | ユーザー自身のブラウザ・IP・セッションを使用 |

**制約:**
- PCが起動していないと動作しない（24時間自動巡回は不可）
- スケーラビリティはユーザー数に依存

### 3.2. 拡張案: ハイブリッド方式

Chrome拡張をベースにしつつ、サーバーサイドの通知機能を追加する折衷案。

- Chrome拡張でレインズUIを強化（フィルタ保存、ワンクリックLINE送信等）
- 条件登録だけサーバーに送り、公開情報（SUUMO等の二次掲載サイト）を監視して速報通知
- **レインズ本体のスクレイピングは一切しない**
- 法的にクリーンかつ速報性も一定担保できる

### 3.3. 棄却案: Browserless + トンネル方式

サーバーサイドでヘッドレスブラウザを動かし、顧客の店舗IPをプロキシとして利用する方式。24時間自動巡回が可能だが、**法的リスクが高く棄却**。

```
[Browserless（ヘッドレスブラウザ管理）]
    ↓
[トンネル（店舗IP経由のプロキシ）]
    ↓
[レインズ]
    ↓
[ラップ型UI（SaaS画面）]
```

**棄却理由:**

| リスク | 深刻度 | 説明 |
|:---|:---|:---|
| **レインズ利用規約違反** | 🔴 高 | スクレイピング・自動アクセスは明確に禁止 |
| **不正アクセス禁止法** | 🔴 高 | CAPTCHA（アクセス制御機能）のAI突破は第2条4項に抵触する可能性 |
| **顧客巻き込み** | 🔴 高 | BANされるのは顧客の店舗IP → 顧客が正規業務不能に → 損害賠償リスク |
| **業界制裁** | 🟡 中 | レインズは国交省指定の公益法人運営。宅建業法上の業務停止処分の可能性 |

### 3.4. アプローチ比較サマリ

| 観点 | Chrome拡張 ✅ | ハイブリッド | トンネル ✕ |
|:---|:---|:---|:---|
| 法的リスク | 🟢 低い | 🟢 低い | 🔴 高い |
| 24時間自動巡回 | ✕ | △（公開情報のみ） | ◎ |
| 開発・運用コスト | 低い | 中 | 高い |
| 差別化ポテンシャル | △ 競合と同質 | ○ 通知で差別化 | ◎ 機能的に最強 |
| **推奨度** | **◎ 推奨** | **○ 検討余地あり** | **✕ 棄却** |

## 4. プラットフォームリスク（中長期）

レインズ側が以下のいずれかを実施した場合、ビジネスが**即死**する可能性がある。

| リスク | 影響度 | 発生確率 |
|:---|:---|:---|
| ブラウザフィンガープリント検知の強化 | 致命的 | 中 |
| アクセスパターンの異常検知 | 致命的 | 中〜高 |
| 二要素認証の導入 | 致命的 | 低〜中 |
| **レインズ公式API提供** | 事業消滅 | 低（だが長期的には可能性あり） |
| 特定業者への見せしめ的処分 | 高 | 低 |

> **核心的な問題**: 生殺与奪をレインズ側に完全に握られている。どんな技術的工夫も、プラットフォーム側の意思決定1つで無効化される。

## 5. PoC計画

### 5.1. MVP（Minimum Viable Product）スコープ

MVPでは、Chrome拡張方式（§3.1）をベースに**既存ツールと同等以上の検索効率化機能**を実装する。差別化機能については§1.3に記載の候補を並行検証する。

#### 確定スコープ（Chrome拡張の基本機能）

| 機能 | 実装内容 | 競合比較 |
|:---|:---|:---|
| 検索条件の保存・復元 | よく使う条件をワンクリックで呼び出し | フォレスト・Miru同等 |
| 新着・変更の差分検知 | 前回検索との差分をバッジ表示 | Miru同等 |
| ワンクリックLINE送信 | 物件情報をLINEで顧客に即共有 | 既存ツールにない |
| UIハイライト・改善 | 検索結果の見やすさ向上 | フォレスト同等 |

#### 検証スコープ（差別化候補・PoC期間内に可否を判断）

| 差別化候補 | 検証内容 | 判断基準 |
|:---|:---|:---|
| 公開情報（SUUMO等）の補助通知 | SUUMOの更新タイミングとレインズとのタイムラグを計測 | レインズより先に情報が取れるケースが一定数あるか |
| 顧客条件マッチング＋自動提案 | Chrome拡張内で顧客リストと条件を管理し、マッチした物件を自動提案 | 現場の仲介担当者にとって実際に使える機能か |
| その他（PoC中に発見したニーズ） | 協力業者へのヒアリングで抽出 | — |

**やらないこと（MVPスコープ外）:**
- 複雑なCRM・顧客管理機能（周辺SaaSと競合するため）
- レインズのサーバーサイド直接スクレイピング（法的・アカウント停止リスク回避）
- 24時間自動巡回（Chrome拡張方式ではPC起動時のみ動作する技術的制約あり）

### 5.2. 前提条件

> **PoCを開始するには、レインズアカウントを持つ協力業者の存在が必須。**

| 前提条件 | 説明 | 難易度 |
|:---|:---|:---|
| **レインズアカウント** | 宅建業免許を持つ不動産会社のみ取得可能。SaaS事業者単体では取得不可 | 🔴 高 |
| **協力業者の確保** | テスト環境としてアカウントを貸してくれる業者が最低1社必要 | 🟡 中 |
| **自社で宅建業免許取得** | 自前で確保する場合、免許取得に2〜3ヶ月+事務所要件あり | 🔴 高 |

**確保パターン:**

| パターン | メリット | リスク |
|:---|:---|:---|
| **A. 羽富が不動産業者** | 即PoC開始可能。顧客パイプラインも期待できる | — |
| **B. 知り合いの業者に協力依頼** | コスト低い | BANリスクを協力者に負わせることになる |
| **C. JVで宅建業免許取得** | 自前でコントロール可能 | 免許取得に時間・コスト（事務所・宅建士確保） |

### 5.5. 資金調達オプション: 株式型クラウドファンディング

株式型CF（FUNDINNO, イークラウド等）を利用して外部資金を調達し、自己負担を軽減する選択肢。

**メリット:**

| メリット | 説明 |
|:---|:---|
| **自己資金の圧縮** | 創業者の出資負担を圧縮可能 |
| **市場検証** | CFの反応自体が市場ニーズのリトマス試験 |
| **初期ユーザー獲得** | 不動産業界の投資家＝潜在的な初期顧客になり得る |
| **PR効果** | メディア露出・認知拡大 |

**リスク・注意点:**

| リスク | 説明 |
|:---|:---|
| **希薄化** | 外部株主が入ることで創業者の持分が薄まる |
| **法人形態の制約** | 株式型CFは**株式会社のみ**対象（合同会社は不可） |
| **情報開示義務** | 事業計画・財務情報を公開する必要がある |
| **事業モデルの露出** | 事業モデルを公にすることで、レインズ側に認識されるリスク |
| **調達コスト** | プラットフォーム手数料（成功報酬15〜20%程度） |
| **株主管理コスト** | 小口株主が多数発生 → 意思決定の遅延リスク |

> ⚠️ **構造的矛盾**: CFで資金調達するには事業モデルを公開する必要があるが、この事業はレインズの規約グレーゾーンで運用するもの。**公開によりレインズ側に認識され、対策を打たれるリスク**がある。

## 🔒 以下、SOCT内部限定メモ

---

### A. SOCTとの分離方針

| 分離項目 | 方針 |
|:---|:---|
| SOCTの屋号・法人格 | **一切使わない** |
| SOCTのエンジニアリソース | **投入しない** |
| SOCTの顧客リスト | **共有しない** |
| 法的責任 | **JV法人に完全に閉じる** |

### B. JV体制・出資構造

| 項目 | 内容 |
|:---|:---|
| 法人形態 | 新設JV（合同会社 or 株式会社） |
| 出資比率 | 羽富 **51%** / 池田 **49%** |
| 代表取締役 | **羽富** |
| 池田の役職 | **なし**（株主のみ） |
| 池田の役割 | 設計・アーキテクチャ監修 + PM補助 |
| 開発体制 | JVがエンジニア1名を直接契約（池田が紹介） |

**池田がJVに役員として入らない理由:**

1. **責任の遮断**: 代表取締役＝羽富 → 法的責任は代表に帰属。池田は善管注意義務を負わない
2. **拒否権の確保**: 49%は特別決議（2/3以上）に対して単独拒否権を行使可能（定款・SHAで確保）
3. **本業への影響ゼロ**: SOCTのRakuco / Tetra LMS / Hanaemiの開発リソースが守られる
4. **レバレッジの最大化**: コードではなく設計力・判断力を提供する関わり方

### C. 池田の工数上限

| 項目 | 上限 |
|:---|:---|
| 月間工数 | **10時間以内**（契約に明記） |
| 対応範囲 | 設計レビュー・アーキテクチャ判断・PM補助 |
| 対象外 | コーディング・顧客対応・営業活動 |

> 「補助」のつもりがフルコミットの技術リードにならないよう、契約で上限を明記する。

### D. 契約で確保すべき条件

**株主間契約（SHA）:**

| 条項 | 内容 |
|:---|:---|
| **拒否権** | 増資・合併・解散・新株発行に対する49%株主の拒否権を明記 |
| **退出条項** | ドラッグアロング / タグアロング権 |
| **競業避止** | SOCTの事業領域との競合を回避する条項 |
| **IP帰属** | 池田が提供した設計・アーキテクチャのIPはJV解散時にSOCTに帰属 |
| **損切り条項** | 6ヶ月PoCの終了条件と清算ルールを明記 |

**エンジニア契約:**

| 項目 | 方針 |
|:---|:---|
| 契約主体 | **JV法人 ↔ エンジニア**の直接業務委託 |

| 理由 | 「SOCTが実質的に運営していた」と認定されるリスクを排除 |
| 紹介者 | SOCTの従業員ではなく、フリーランスとして独立しているエンジニア |
