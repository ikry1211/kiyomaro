## 1. システムの全体像と目的

「Kiyomaro（清麿）」は、株式会社SOCTにおける次世代の **経営OS（自律型経営戦略AI基盤）** です。
単なる静的なドキュメント管理システムではなく、AI（Kiyomaro）が経営戦略の立案・更新・実行を動的にサポートし、経営陣の思考を拡張するパートナーとして機能することを目的とします。

### 1.1. コア設計思想

本システムの根幹にあるのは、以下の2つの原則です。

1. **戦略のバージョン管理（Event Sourcing 的アプローチ）**
   - 経営戦略ドキュメントは Git やファイルシステムではなく、**Supabase (PostgreSQL) 上で厳密にバージョン管理**されます。
   - ドキュメントの直接的な上書きは行わず、すべての変更は「追記（INSERT）」と「状態変更（UPDATE state）」によって管理されます。
   - これにより、「誰が・いつ・なぜ・何を変更したか」の完全な監査証跡（Audit Trail）を保持できます。

2. **AIによる提案と人間による承認（Human-in-the-loop）**
   - AIが直接的に戦略を書き換えることはありません。
   - AIは常に「提案（proposed）」状態のドラフトを作成し、最終的な適用は必ず人間（経営陣）の承認を経て行われます。

## 2. モノレポ構成（Turborepo）

本プロジェクトは **Turborepo** を用いたモノレポとして構成されます。

```
kiyomaro/
├── apps/
│   ├── web/          ... Next.js アプリケーション本体（フロントエンド / BFF）
│   └── docs/         ... システム仕様書・設計ドキュメント（本ファイル群）
├── packages/
│   └── supabase/     ... Supabase 共通基盤（マイグレーション、型定義、シードデータ等）
├── turbo.json
└── package.json
```

| ディレクトリ | 役割 |
|:---|:---|
| `apps/web` | ユーザー向けWebアプリケーション。Next.js (App Router) で構築し、Vercel にデプロイ。 |
| `apps/docs` | 開発チーム向けのシステム仕様書・設計ドキュメント管理。 |
| `packages/supabase` | DB マイグレーション、RLS ポリシー、TypeScript 型定義の共有パッケージ。 |

## 3. コンポーネント構成と役割分担

本システムは、以下の3つの主要コンポーネントで構成されます。

### 3.1. フロントエンド / BFF（Backend For Frontend）

- **技術スタック:** Next.js (App Router)
- **デプロイ先:** Vercel（東京リージョン `hnd1` を必須指定）
- **役割:**
  - ユーザーインターフェースの提供（SSR / RSC による高速な画面描画）
  - API Routes / Server Actions を通じた BFF 層としての機能
  - Vercel AI SDK を用いた AI プロバイダー（Gemini API）との通信ハブ
  - Supabase Auth と連携した認証・認可の統合
- **設計方針:**
  - デフォルトは Server Components を使用し、データ取得はサーバー側で実行
  - `"use client"` は必要最小限のインタラクティブコンポーネントにのみ適用
  - UIとビジネスロジックの関心の分離を徹底

### 3.2. AI 連携・オーケストレーション

- **技術スタック:** Vercel AI SDK, Gemini API
- **役割:**
  - AI 人格「Kiyomaro」としての振る舞いを実装
  - `kiyomaro_settings` テーブルから動的にシステムプロンプトやパラメータを取得し、AIの挙動を制御
  - ユーザーからのインプット（課題・アイデア）と既存の戦略ドキュメント（`active` 状態）を元に、新バージョンのドラフトを生成
  - ストリーミングレスポンスによるリアルタイムなテキスト生成体験の提供

### 3.3. バックエンド / データベース

- **技術スタック:** Supabase (PostgreSQL)
- **役割:**
  - 戦略ドキュメント、バージョン履歴、AI設定データの永続化
  - Row Level Security (RLS) を用いたセキュアなデータアクセス制御
  - Supabase Auth を用いた認証基盤
  - DB トリガーやファンクションによるビジネスルールの強制（例: 同一 slug に active は最大1つ）

## 4. AI（Kiyomaro）による「戦略の更新提案フロー」

Kiyomaro の最大の特徴は、ドキュメントを直接上書きするのではなく、**提案 → レビュー → 承認** という明確なワークフローを通じて戦略を進化させる点にあります。

### 4.1. フロー概要図

```
┌─────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  ユーザー    │    │   Kiyomaro (AI)   │    │   Supabase (DB)  │
│  (経営陣)    │    │  Gemini API       │    │   PostgreSQL     │
└──────┬──────┘    └────────┬─────────┘    └────────┬─────────┘
       │                    │                       │
       │ 1. 課題・アイデアを │                       │
       │    インプット       │                       │
       │ ──────────────────>│                       │
       │                    │ 2. 現在の active 版を  │
       │                    │    取得                │
       │                    │ ─────────────────────>│
       │                    │ <─────────────────────│
       │                    │                       │
       │                    │ 3. 新バージョンの      │
       │                    │    ドラフトを生成       │
       │                    │                       │
       │                    │ 4. proposed として     │
       │                    │    INSERT              │
       │                    │ ─────────────────────>│
       │                    │                       │
       │ 5. 差分レビュー画面 │                       │
       │ <──────────────────│                       │
       │                    │                       │
       │ 6a. 承認           │                       │
       │ ──────────────────────────────────────────>│
       │    (active→archived, proposed→active)      │
       │                    │                       │
       │ 6b. 棄却 or 修正依頼│                       │
       │ ──────────────────>│                       │
       │                    │ → 再ドラフト or 破棄   │
```

### 4.2. 各ステップの詳細

| Step | アクション | 詳細 |
|:---:|:---|:---|
| 1 | **ユーザーのインプット** | ユーザーが現在の戦略に対する課題、新たなアイデア、外部環境の変化などをKiyomaroにインプットする。 |
| 2 | **現行戦略の取得** | Kiyomaroが対象ドキュメント（`slug`）の `state = 'active'` レコードをDBから取得し、コンテキストとして利用する。 |
| 3 | **AIによるドラフト作成** | Gemini API がインプットと現行戦略を踏まえ、新しいバージョンのドキュメント（ドラフト）を生成する。 |
| 4 | **提案データの保存** | 生成されたドラフトは既存データを上書きせず、同一 `slug` に対して新しい `version` 番号と `state = 'proposed'` として INSERT される。 |
| 5 | **人間によるレビュー** | ユーザー（経営陣）は、現在の `active` 版と `proposed` 版の差分を画面上で比較・レビューする。 |
| 6a | **承認・昇格** | 承認時：既存の `active` レコードを `archived` に変更し、承認された `proposed` レコードを `active` に昇格させる。 |
| 6b | **棄却・修正依頼** | 提案が要件に合わない場合、ユーザーは追加フィードバックを与えて再ドラフトを依頼、または提案を破棄する。 |

### 4.3. このフローの価値

- **完全な監査証跡**: 「誰が・いつ・なぜ戦略を変更したのか」「AIがどのような提案を行ったか」の履歴を完全に保持
- **安全なAI活用**: AIが暴走して戦略を勝手に書き換えるリスクをゼロにする
- **段階的な進化**: 戦略の変遷を時系列で辿ることができ、意思決定の振り返りに活用可能

## 5. AI ツール仕様（Function Calling）

Kiyomaro は Vercel AI SDK の `tools`（Function Calling）を用いて、外部データソースやデータベースと自律的に連携します。
以下の3つのコアツール（関数）を定義し、`apps/web/app/api/chat/route.ts` 等で Gemini API に渡す `tools` として実装します。

### 5.1. ツール一覧

| ツール名 | 役割 | 権限 | 実装フェーズ |
|:---|:---|:---|:---|
| `manage_strategy_db` | 戦略DBアクセス | Read + Write（制限付き） | Phase 1 |
| `read_financial_sheets` | 予実データ読み取り | **Read-Only** | Phase 1 |
| `search_past_context` | RAG連携（過去文脈検索） | **Read-Only** | Phase 2 |

---

### 5.2. `manage_strategy_db`（戦略DBアクセスツール）

- **役割:** Supabase の `strategy_documents` テーブルへのアクセスを提供する。
- **連携先:** Supabase (PostgreSQL)
- **権限:**

| 操作 | 許可 | 詳細 |
|:---|:---:|:---|
| Read | ✅ | 現在の `state = 'active'` な戦略ドキュメントを取得する。過去の `archived` 版の参照も可能。 |
| Write (INSERT) | ✅ | 既存レコードを**上書きせず**、新しい `version` で `state = 'proposed'` としてレコードを INSERT する。 |
| Write (UPDATE) | ❌ | 既存レコードの `state` 変更（承認・アーカイブ）は**人間の操作でのみ実行**。AIには許可しない。 |
| DELETE | ❌ | 削除は禁止。 |

- **設計意図:** AIが戦略を「提案」することはできるが、「適用」することはできない。Human-in-the-loop の原則を技術的に強制する。

---

### 5.3. `read_financial_sheets`（予実データ読み取りツール）

- **役割:** Google Sheets API を経由し、SOCT社の「予実管理スプレッドシート」から最新の財務数値（Fact）を取得する。
- **連携先:** Google Sheets API
- **権限:** **完全な Read-Only**。スプレッドシートの書き換えは絶対に行わない。
- **実装フェーズ:** Phase 1（戦略DBアクセスと並行して実装）

| 操作 | 許可 | 詳細 |
|:---|:---:|:---|
| Read | ✅ | 指定されたスプレッドシートから特定のシート・セル範囲のデータを取得する。 |
| Write | ❌ | 一切の書き込みを禁止。API キーのスコープも `readonly` に制限する。 |

- **ユースケース:**
  - Kiyomaro が戦略提案を作成する際、最新の売上実績・予算消化率などの定量データを根拠として参照する。
  - 「現在の財務状況を踏まえて中期計画を見直して」といったユーザーの指示に対応する。

---

### 5.4. `search_past_context`（RAG連携ツール）

- **役割:** 過去の議事録や経営会議の文脈を検索するため、独立した Python RAG マイクロサービスの API を呼び出す。
- **連携先:** Python RAG マイクロサービス（別途構築、Phase 2）
- **権限:** **Read-Only**。検索クエリの送信と結果の受信のみ。
- **実装フェーズ:** Phase 2

| 操作 | 許可 | 詳細 |
|:---|:---:|:---|
| Search / Read | ✅ | 検索クエリを送信し、関連するドキュメントチャンクを取得する。 |
| Write | ❌ | RAG インデックスへの書き込み・更新は行わない。 |

- **ユースケース:**
  - 「前回の取締役会で決まった方針を踏まえて」といった文脈依存の指示に対応する。
  - 過去の議事録から関連する意思決定の経緯を自動的に取得し、提案の精度を高める。
