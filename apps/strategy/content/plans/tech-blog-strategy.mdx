## 3. 核心: 開発の副産物から自動生成する

### 3.1. 従来のテックブログとの違い

| | 従来のテックブログ | **SOCTの自動生成モデル** |
|:---|:---|:---|
| **執筆** | エンジニアが業務時間を割いて書く | 🤖 開発の副産物（Git diff・会話ログ）からAIが生成 |
| **継続性** | 月1未満に落ちがち → 放置 → 逆効果 | 開発が続く限りネタが自動で溜まる |
| **池田の工数** | 月2〜4時間（執筆） | **月30分〜1時間（レビュー・承認のみ）** |
| **正確性** | 記憶ベースで書く → 不正確になりがち | ソースコードと会話ログが根拠 → 正確 |

### 3.2. パイプライン設計

```
【入力】開発の副産物（すでに存在する）
  ├── 📝 Git diff / コミット履歴（何を作ったか）
  ├── 💬 AIとの会話ログ（なぜその設計にしたか）
  ├── 📄 PR / Issue（決定事項のサマリー）
  └── 📋 仕様書（apps/docs/）
      ↓
【処理】LLM（Gemini / Claude）
  ├── 開発内容の要約
  ├── 技術的な意思決定の抽出
  ├── 読者向けのストーリー構成
  └── MDX形式で出力
      ↓
【レビュー】池田（30分〜1時間）
  ├── 事実確認
  ├── 機密情報の除去
  └── トーンの調整
      ↓
【公開】blog.soct.jp
```

### 3.3. YouTubeパイプラインとの関係

[YouTube自動編集パイプライン](./youtube-automation) と**入力素材が共通**。

```
                    Git diff
                    会話ログ
                    録画
                      │
            ┌─────────┼─────────┐
            ↓                   ↓
     テックブログ          YouTube動画
     パイプライン          パイプライン
   ┌───────────┐      ┌───────────┐
   │ LLM要約    │      │ Gemini解析 │
   │ MDX生成    │      │ Remotion   │
   │ → テキスト  │      │ FFmpeg     │
   └───────────┘      │ → 動画     │
                      └───────────┘
```

テックブログの方が圧倒的にシンプル（LLM APIだけで完結）。先行して素材収集の知見を蓄積し、YouTube構築時に共通化する。

## 4. コンテンツ設計

### 4.1. 読者

**エンジニアではなく、意思決定者。**

| 読者 | 求めている情報 | ブログで伝えること |
|:---|:---|:---|
| **受託を発注検討中の事業責任者** | この会社に頼んで大丈夫か | AI伴走型で高品質・高速に作れる実績 |
| **広告運用を検討中の企業** | 他の代理店と何が違うか | AI活用で従来型より効率的 |
| **Tetra LMS見込みテナント** | 技術基盤は信頼できるか | プラットフォームの設計思想・セキュリティ |
| **M&A買い手候補** | 技術力の評価 | 開発力・AI活用度の可視化 |
| **将来の採用候補（エンジニア）** | どんな開発をしている会社か | 技術スタック・開発文化 |

### 4.2. 記事のトーン

| ❌ やらない（エンジニア向け） | ✅ やる（意思決定者向け） |
|:---|:---|
| 「Next.js App Routerのキャッシュ戦略を解説」 | 「**1人のPdMとAIで研修プラットフォームを作った話**」 |
| 「SupabaseのRLSでマルチテナントを実装する方法」 | 「**なぜ少人数×AIで高品質な開発ができるのか**」 |
| 「Remotionでプログラマティック動画生成」 | 「**テレアポ1,266件の通話をAIで分析して見えたこと**」 |

**「何を・なぜ・どうやった」のストーリーが軸。** コードの詳細は補足程度。

### 4.3. コンテンツカテゴリ

| カテゴリ | 内容 | ソース | 頻度 |
|:---|:---|:---|:---|
| **🏗️ ビルドログ** | 開発プロジェクトの記録 | Git diff + 会話ログ + 仕様書 | 月1〜2本 |
| **🤖 AI活用事例** | AIとの協業プロセスの紹介 | 会話ログ + PR | 月1本 |
| **📊 技術選定** | 意思決定の記録（ADR） | 設計ドキュメント | 不定期 |
| **🔧 運用知見** | SaaS運用のノウハウ | 運用ログ | 不定期 |

### 4.4. 書かないこと

- Reactチュートリアル的な技術解説（読者が違う）
- 顧客の機密に触れる内容（守秘義務）
- 低品質・中途半端な記事（出さない方がマシ）

### 4.5. 記事ネタの評価軸

| 軸 | 問い | 判断基準 |
|:---|:---|:---|
| **① 機密リスク** | 出して大丈夫か？ | 料金体系・顧客名・戦略の核心が含まれていないか |
| **② 独自性** | SOCTにしか書けないか？ | 汎用的な技術解説は誰でも書ける。自社の実体験・実数値は唯一無二 |
| **③ 具体性** | 数字やビフォーアフターがあるか？ | 「AIで効率化」は弱い。「1,266件を分析し◯%改善」は強い |
| **④ ストーリー性** | 読んで面白いか？ | 意思決定の葛藤・失敗・意外な結果がある記事は読まれる |
| **⑤ ブランド貢献** | 「SOCTに相談しよう」と思うか？ | 「すごい」で終わらず「この会社と組みたい」と思わせるか |

> **迷ったときの判断基準:** 「これを読んだ中川さんが、もっと早くSOCTに連絡しようと思うか？」

## 7. 自動生成パイプラインのフェーズ

### Phase 1: 手動 + LLM（今すぐ開始可能）

```
① 素材を選ぶ（Git diff / 会話ログ / 仕様書）
② LLMに投入 → 記事ドラフト生成
③ 池田がレビュー → 修正 → 公開
```

ツール構築不要。LLMとの対話だけで記事が生成できる。

### Phase 2: 半自動化

```
① Git hooks / CI で素材を自動収集
② 週次 or 月次で記事ドラフトを自動生成 → Slack通知
③ 池田がレビュー → git push → 自動デプロイ
```

### Phase 3: YouTube連動（パイプライン統合）

同じ素材からテックブログ記事・YouTube動画・Xポストを同時生成する統合パイプライン。
