## 1. ゴール設定

| 目的 | 区分 | 内容 |
|:---|:---|:---|
| **主目的** | パイプライン技術の実証 | 動画自動編集パイプラインの実用データを得る |
| **主目的** | 楽しさ・発信 | ゲーム/ものづくり自体を楽しむ。持続可能性の源泉 |
| **副次効果** | 技術ブランディング | AI活用・自動化の実績として対外的にアピール |
| **長期目標** | 収益化 | 軌道に乗ればAdSense収益を法人売上として計上 |

### マイルストーン

| 時期 | 目標 | 具体的指標 |
|:---|:---|:---|

> 技術系チャンネルは初速が遅いが、コンテンツが積み上がるとロングテールで再生が伸びる。焦らないことが重要。

## 2. チャンネル構成: 2チャンネル体制

**パイプラインは1つ、テンプレートとアップロード先だけ分ける**構成。

| | CH1: ブラウザゲーム開発 | CH2: 3Dプリンタ ものづくり |
|:---|:---|:---|
| **テーマ** | AIと協業してブラウザゲームを作る | AIと議論しながらコードで3D設計 → 3Dプリントで実体化 |
| **制作手段** | コード（JS/TS/Canvas/WebGL） | コード（OpenSCAD/CadQuery等） |
| **素材の性質** | ゲーム画面キャプチャ | コード + 3Dプレビュー + 実写 |
| **視聴者体験** | **URL1つで即プレイ可能** | STLファイル公開で再現可能 |
| **開始優先度** | ✅ **先行** | 横展開（パイプライン確立後） |

> **チャンネルを分ける理由**: YouTubeのアルゴリズムはテーマ一貫性を重視。混在すると「何のチャンネルか」が不明になりレコメンドされにくくなる。

## 4. 映像構成: エディタを映さない方針

### 方針

エディタ（VS Code等）やターミナルの画面は**原則として映さない**。

### 理由

| 理由 | 詳細 |
|:---|:---|
| **セキュリティ** | 環境変数・APIキー・パスワード・メールアドレス・ファイルパスの漏洩リスクをゼロにする |
| **視聴体験** | コードをタイピングしている画面は退屈。AIの応答待ちも退屈 |
| **自動化との相性** | Git diffからRemotionで自動生成できるため、録画すら不要 |
| **編集の自由度** | 後からコードの見せ方を調整できる |

### 動画に映すもの

| 要素 | 取得方法 | 映像化手段 |
|:---|:---|:---|
| **ゲーム画面（デモ）** | Playwright で自動録画 | 自動生成 |
| **AIテストプレイ** | AIが操作コード生成 → Playwright実行 → Canvas APIで録画 | 自動生成（コンテンツとしても面白い） |
| **AIとの設計議論** | チャットログ（テキスト） | Remotionで再構成（チャットUI風アニメーション） |
| **コードの変化** | Git diff | Remotionでシンタックスハイライト付きアニメーション |
| **設計図・仕様** | テキスト/マークダウン | Remotionで図解・モーション生成 |
| **ナレーション** | TTS（Gemini Flash） | 音声トラック |

> **OBSは不要。** ブラウザゲームはCanvas API（`canvas.captureStream()`）やPlaywrightで**プログラマティックに録画可能**。録画自体もパイプラインの一部として完全自動化できる。

### 動画に映さないもの

| 要素 | 理由 |
|:---|:---|
| ❌ VS Code / エディタ画面 | セキュリティリスク + 退屈 |
| ❌ ターミナル | 環境変数・パス漏洩リスク |
| ❌ ブラウザDevTools | トークン・Cookie漏洩リスク |
| ❌ デスクトップ全画面 | 通知・ブックマーク等の漏洩リスク |

### AIテストプレイ

完成したゲームを**AIに遊ばせる**。これ自体がコンテンツとして面白い。

```
ゲーム完成 → AIにコードを渡す → AIがPlaywright操作コードを生成 → 自動プレイ + 録画
```

| AIの振る舞い | 視聴者の反応 |
|:---|:---|
| 上手くプレイする | 「すごい、ちゃんと動いてる！」 |
| 変な動きをする | 「www」（失敗もエンタメ） |
| バグを発見する | 修正パートに繋がる → 構成が自然に膨らむ |

### 動画1本の構成テンプレート

```
1. 🎯 完成ゲームのデモ（30秒）
   → Playwright自動録画: ゲーム画面

2. 🤖 AIとの設計議論ダイジェスト（2分）
   → Remotion生成: チャットログを要約してアニメーション化

3. 💻 コード解説（3分）
   → Remotion生成: Git diffをシンタックスハイライト付きで表示
   → ナレーション: TTS

4. 🎮 AIテストプレイ（1〜2分）
   → Playwright + Canvas API: AIが自動プレイ + 録画
   → 上手くいっても失敗しても面白い

5. 🐛 バグ修正（あれば）（1分）
   → AIテストで見つかった問題の修正過程

6. 📊 まとめ・スタジオのタイトル一覧（1分）
   → Remotion生成: テンプレート
```

### 動画テンプレート詳細（分単位）

想定尺: **8〜10分**

| 時間 | パート | 内容 | 素材 |
|:---|:---|:---|:---|
| 0:00 | **フック** | 完成ゲームのハイライト・最も映える瞬間 | Playwright録画 |
| 0:10 | **オープニング** | スタジオロゴ + タイトルカード「宇都宮ゲームスタジオ #001」| Remotionテンプレ |
| 0:30 | **ゲーム紹介** | 完成デモプレイ + 操作方法テロップ | Playwright録画 |
| 1:30 | **AIとの設計議論** | チャットログから面白い議論をダイジェスト | チャットログ → Remotion |
| 4:00 | **コード実装** | Git diffをステップごとに表示 + 解説 | Git diff → Remotion |
| 6:30 | **AIテストプレイ** | AIが自動プレイ。実況風ナレーション | Playwright + Canvas API |
| 8:00 | **バグ修正** | AIテストで見つかった問題の修正（あれば） | Git diff → Remotion |
| 8:30 | **エンディング** | プレイリンク告知 + スタジオタイトル一覧更新 | Remotionテンプレ |

## 5. 素材の取得タイミング

エディタを映さない方針なので、**素材はすべて開発の副産物として自動的に生まれる**。追加の撮影作業はほぼ不要。

### フェーズ別の素材一覧

#### Phase A: AIとの設計議論

| 素材 | 保存方法 | 意識すること |
|:---|:---|:---|
| **チャットログ** | チャットツールからエクスポート | **最重要**。議論ログが動画の核 |

→ **1トピック1ファイル**で分けておくと、動画編集時にどの議論をピックアップするか選びやすい。

#### Phase B: コーディング

| 素材 | 保存方法 | 意識すること |
|:---|:---|:---|
| **Git diff** | 普通にcommitするだけ | **こまめにコミット**する（動画のステップ数に直結） |

コミットの粒度 = 動画のコード解説のステップ数。各コミットがアニメーションで1ステップになる。

```
❌ 悪い例: "ゲーム完成"（1コミットに全部）
✅ 良い例:
  "Canvas初期化とゲームループ作成"
  "パドルの描画と操作"
  "ボールの移動と壁反射"
  "ブロックの配置と衝突判定"
  "スコア表示とゲームオーバー"
  "難易度調整とエフェクト追加"
```

#### Phase C: 途中の動作確認（任意）

| 素材 | 保存方法 | 意識すること |
|:---|:---|:---|
| 途中スクリーンショット | Playwright or 手動 | あると「ビフォーアフター」で使える |
| バグの記録 | コミットメッセージ or メモ | 面白いバグは動画ネタになる |

> オプション: 主要コミットごとにPlaywrightでスクショを自動取得する `post-commit` フックを入れると、進捗アニメーションが自動生成できる。

#### Phase D: 完成・仕上げ

| 素材 | 保存方法 | 意識すること |
|:---|:---|:---|
| 完成版コード | `git tag v1.0` | タグを打っておく |
| デプロイURL | Vercelデプロイ | 動画説明欄に載せる |

#### Phase E: テストプレイ・録画（自動）

| 素材 | 保存方法 |
|:---|:---|
| 完成デモ映像 | Playwright自動録画スクリプトを実行 |
| AIテストプレイ映像 | AI生成操作 + Playwright実行 |

### 制作中のチェックリスト

ゲームを作るとき、これだけ意識すればOK:

| # | タイミング | やること | 自動? |
|:---|:---|:---|:---|
| 1 | AI議論開始時 | チャットログの保存先を決める | 手動（初回のみ） |
| 2 | 議論中 | 1トピック1ファイルで保存 | 手動 |
| 3 | コーディング中 | **こまめにコミット** | 手動（習慣化） |
| 4 | コミット時 | コミットメッセージを丁寧に | 手動 |
| 5 | バグ発見時 | 面白いバグはメモ | 手動（任意） |
| 6 | ゲーム完成 | `git tag v1.0` + Vercelデプロイ | 手動 |
| 7 | 完成後 | Playwright録画スクリプト実行 | ✅ 自動 |
| 8 | 完成後 | AIテストプレイスクリプト実行 | ✅ 自動 |

> 開発中に追加で必要な作業は「**チャットログを保存する**」と「**こまめにコミットする**」の2つだけ。

### 素材の保存先（game-studio 内）

すべての素材は `game-studio/` リポジトリ内に保存される（プロジェクト構成の全体像は §7 を参照）。

```
game-studio/
├── docs/
│   └── chat-logs/               ← AIとの議論ログ（動画素材①）
│       ├── td-01-design.md      （td = tower-defense）
│       ├── td-02-balance.md
│       └── ...
├── recordings/                  ← 自動録画の出力先（動画素材②）
│   ├── tower-defense-demo.webm
│   └── tower-defense-ai-playtest.webm
└── .git/                        ← コミット履歴（動画素材③）
```

> 動画のレンダリング出力（final.mp4 等）は別プロジェクト `youtube-pipeline` 側で生成される。game-studio には含めない（§8 参照）。

### エージェントルールによる自動化

上記のチェックリスト（コミット粒度・チャットログ保存等）は**人間が意識するのではなく、ゲーム開発を担当するAIエージェントのルール（GEMINI.md）として組み込む**。

| ルール | エージェントがやること |
|:---|:---|
| **コミット粒度** | 1機能1コミット。「修正」「WIP」等の曖昧なメッセージ禁止 |
| **チャットログ保存** | 設計議論を `docs/chat-logs/` に1トピック1ファイルで自動保存 |
| **ディレクトリ構造** | §7 のプロジェクト構成を自動的に維持 |
| **バグ記録** | 面白いバグや予想外の挙動を `docs/chat-logs/bugs.md` に自動追記 |
| **リリースタグ** | ゲーム完成時に `git tag vX.Y` を自動実行 |
| **README生成** | ゲーム概要・操作方法・プレイURLを自動記載 |

> 詳細はゲーム開発リポジトリ立ち上げ時に `GEMINI.md` として策定する。

## 6. コンテンツ戦略: ハイブリッド構成

### タイトルの基準

インディーゲームスタジオとして、テックデモではなく**「遊んで面白い」タイトル**をリリースする。

| 要素 | 基準 |
|:---|:---|
| プレイ時間 | 最低5分は遊べる |
| リプレイ性 | 「もう1回」がある |
| 進行感 | スコア / Wave / レベル等の成長要素 |
| 演出 | SE・エフェクト・画面揺れ等のジューシーさ |
| スマホ | タッチ操作対応 |

### シーズン1（1〜3ヶ月目）: 単発ゲーム集

1動画1ゲーム完結で量産。パイプライン検証 + テンプレート確立 + 視聴者層を掴む。

| # | ゲーム | ジャンル | 中毒性の鍵 | 開発規模 |
|:---|:---|:---|:---|:---|
| **001** | **タワーディフェンス** | ストラテジー | タワー配置の試行錯誤。Wave制 | ★★☆ |
| 002 | サバイバル（VampSurv風） | アクション | 自動攻撃 + アップグレード選択 | ★★☆ |
| 003 | ローグライクダンジョン | RPG | ランダム生成 + 探索 | ★★★ |
| 004 | リズムゲーム | 音楽 | Web Audio API。感覚的操作 | ★★☆ |
| 005 | デッキビルダー | カード | 毎回違うビルド | ★★★ |
| 006 | 弾幕STG | シューティング | パターン暗記 + スコアアタック | ★★☆ |

> 毎回の動画で完成品が遊べる → 説明欄のリンクから即プレイ → エンゲージメント向上

### 第1弾: タワーディフェンス 詳細設計

FF7フォートコンドルのような「シンプルだが戦略性の高い」TDを目指す。

#### 最小構成（v1.0）

| 要素 | 内容 |
|:---|:---|
| ステージ | 1マップ（蛇行パス） |
| 操作 | クリック/タップでタワー設置 |
| タワー | 3種（射撃塔 / 減速塔 / 範囲攻撃塔） |
| 敵 | 3種（ザコ / 速い / 固い） |
| Wave | 10〜15Wave。全Wave突破でクリア |
| 経済 | 敵撃破で資金 → 新タワー or アップグレード |

#### コミットステップ（= 動画の解説ステップ）

```
 1. "マップのグリッド描画とパス定義"
 2. "敵のスポーンとパスに沿った移動"
 3. "タワーの設置UIと配置制限"
 4. "射撃塔の実装（最寄りの敵を攻撃）"
 5. "敵撃破と資金システム"
 6. "減速塔と範囲攻撃塔の追加"
 7. "Wave制と難易度カーブ"
 8. "タワーのアップグレードシステム"
 9. "拠点HP・ゲームオーバー・クリア判定"
10. "エフェクト・SE・UI磨き込み"
11. "スマホ対応（タッチ操作）"
```

#### 動画としての見どころ

| パート | 内容 |
|:---|:---|
| AI議論 | 「タワーのバランスどうする？」「Wave設計の曲線は？」 |
| コード解説 | パスファインディング、射程と優先ターゲット |
| AIテストプレイ | AIが最適配置を模索。人間より上手いか？ |
| バグ修正 | 敵がタワーをすり抜ける等の定番バグ |

### シーズン2（4ヶ月目〜）: 大型プロジェクト

シーズン1で掴んだ視聴者を長期シリーズで囲い込む。

```
例: 「AIと100日でブラウザRPGを作る」
  Episode 1: 世界観設計 + マップ表示
  Episode 2: キャラクター移動 + 衝突判定
  Episode 3: 戦闘システム
  ...
```

> シーズン2が始まると、シーズン1の動画が「入門用コンテンツ」として機能し、新規→過去動画の回遊。

### 並行

マンネリ防止と新規流入のため、シーズン2期間中も息抜きの単発ゲームを月1本程度挟む。

### ショート動画戦略

ショート動画は**長尺動画の完成後に自動切り出し**で生成する。制作中に別途ショートを撮る運用はしない。

#### ショートの役割

```
ショート（認知・新規流入） → チャンネルページ → 長尺動画（ファン定着）
                                             → ゲームポータル（プレイ）
```

#### ショートに切り出すコンテンツ

| 種類 | 内容 | 尺 |
|:---|:---|:---|
| **ゲームハイライト** | 完成ゲームの最も映えるシーン | 15〜30秒 |
| **AIテストプレイ** | AIの面白い動き・バグ遭遇シーン | 15〜30秒 |
| **ビフォーアフター** | Step1 → Step10 の進化を早送り | 15秒 |
| **AIとの会話** | 設計議論の面白い一言を抜粋 | 15〜30秒 |

> 長尺1本 → ショート3〜5本を自動生成。追加の制作コストはほぼゼロ。

#### 投稿スケジュール例

```
月曜: 長尺動画 公開
火曜: ショート① ゲームハイライト
水曜: ショート② AIテストプレイの面白シーン
金曜: ショート③ ビフォーアフター
```

> **ティーザー（開発中チラ見せ）はX（@IkePgame）で運用**。チャンネルが成長してきたらショートにも展開を検討。

## 7. ゲームポータル設計

### プロジェクト構成

ゲームは**1リポジトリ（Next.js）にまとめてポータルサイトとして運用**する。itch.ioのような「スタジオ公式ゲームカタログ」を自前で持つ。

```
game-studio/                    ← 1リポジトリ（Next.js）
├── app/
```

- Vercel 1プロジェクトで完結。デプロイも1箇所。
- **録画・テストの自動化**: `scripts/e2e-record/` 内のPlaywrightスクリプトを走らせることで、Headlessブラウザがポータルの各ゲームにアクセスし、AIテストプレイを実行しつつ動画（webm/mp4）をローカルに吐き出す仕組み。
- **ゲーム側での録画**: `useGameRecorder` などのHooksを用意してコンポーネント側に組み込むことで、手動・自動問わず「最も映えるシーン」をブラウザ内で完結して録画できるようにする。

### 認証戦略: ゲストプレイ + 任意ログイン

**ログインしなくても即プレイ可能。ログインするとスコア永続化・ランキング登録ができる。**

#### ログイン方式

| 方式 | 優先度 | 理由 |
|:---|:---|:---|
| **Googleログイン** | ◎ メイン | ほぼ全員が持っている。ワンクリック |
| **Xログイン** | ○ サブ | @IkePgameとの連携。シェア導線 |
| ゲスト（匿名） | ◎ デフォルト | ログイン不要で即プレイ |

> メール+パスワードは登録の手間が大きいため不採用。

#### 認証基盤: Supabase Auth

Tetra LMSと同じ技術スタックで統一。

#### ユーザー体験の流れ

```
1. YouTubeからリンクをクリック
2. 即プレイ開始（ゲスト。ログイン不要）
3. ゲーム中にスコア記録
4. ゲームオーバー時:
   「スコアをランキングに登録する？」
   → [Googleでログイン] [Xでログイン] [登録しない]
5. ログイン → ローカルスコアがアカウントに紐づく
6. 次回以降は自動ログイン状態でプレイ
```

> ゲーム中に邪魔せず、**「スコアを残したい」という自然な動機**でログインしてもらう。

## 8. 動画制作パイプラインとの境界線（インターフェース）

ゲームリポジトリは動画編集等の重い処理を持たず、**「動画生成パイプラインへ渡すための生素材を出力すること」**に専念する。動画のレンダリング（Remotion/FFmpeg）は別プロジェクト空間で実行する。

### ゲーム側が出力すべき3つの連携仕様（素材）

| 素材 | 拡張子 | 保存先 / 出力元 | 用途（パイプライン側での扱い） |
|:---|:---|:---|:---|
| **① プレイ録画映像** | `.mp4` または `.webm` | `recordings/` | 冒頭デモのハイライト、AIテストプレイの映像として無加工で使用 |
| **② AIとのチャットログ** | `.md` | `docs/chat-logs/` | Gemini APIに食わせ、「ナレーション台本」と「AI議論アニメーション」を自動生成する |
| **③ コミット履歴** | git commit history | `.git/` | Remotion が履歴を読み取り、「コードがステップごとに書き変わるアニメーション」を生成する |

### データフロー

```
【入力】game-studio（ゲームリポジトリ）
  ├─ 🎮 ゲームの機能本体
  ├─ 🎥 録画データ (E2Eで自動録画)
  ├─ 💬 チャットログ
  └─ 📝 Git diff履歴
      ↓ 
    (CLIコマンド等で別パイプラインへ渡す)
      ↓
【出力】youtube-pipeline（動画生成側）
  ├─ 🪄 Gemini API（シナリオ出力）
  ├─ 🎬 Remotion（アニメーション合成）
  └─ 🎞️ FFmpeg（最終動画出力）
```

> **運用における人間（いけ氏）の作業:** 
> 「AIと一緒にゲーム本体を開発し、テストスクリプトを走らせて完成させる」だけで終わる。その副作用として①②③が自動で溜まり、動画パイプラインへ渡せる状態になる。

## 9. 台本・制作ワークフロー

### 台本の粒度: 3段階

このチャンネルは「制作過程がそのままコンテンツ」なので、台本は**録画後に後付け**が基本。

| レベル | タイミング | 内容 | 例 |
|:---|:---|:---|:---|
| **L1: 企画メモ** | 制作前 | 「何を作るか」だけ。1〜3行 | 「Canvas APIで弾幕STGを作る」 |
| **L2: 構成アウトライン** | 録画前 | 動画の大まかな章立て。5〜8項目 | ①完成デモ ②設計方針 ③実装 ④テスト |
| **L3: ナレーション台本** | **録画後** | Gemini解析結果を基にTTS用テキスト作成 | 各シーンのナレーション全文 |

> L3を録画後に作る理由: AIとの議論は予測できず、事前台本通りにならない。素材を見てから構成を決める方が自然で品質も高い。

### 制作方式: バッチ制作 → 週1公開

| フェーズ | 内容 |
|:---|:---|
| **制作フェーズ（月1〜2日）** | 3〜4本分のゲーム制作を集中して行う |
| **素材自動生成** | ゲーム画面の自動録画 + AIテストプレイ + Git diff + チャットログ |
| **パイプライン処理** | Gemini解析 → FFmpeg → Remotion を3本分バッチ実行 |
| **人間レビュー** | 解析結果の確認・ナレーション台本作成 |
| **公開フェーズ（週1本）** | ストックから順次公開。コメント対応 |

| 指標 | 目標値 |
|:---|:---|
| 投稿頻度 | **週1本**（最低ライン） |
| 制作集中日数 | **月1〜2日** |
| 常時ストック | **2〜3本先行** |

## 13. 収益性（YouTube ＋ ブラウザゲーム）

外部のゲームプラットフォーム（PokiやCrazyGames等）には意図的に配信せず、**自社ポータル（game-studio）に全トラフィックを集中させて「独自のスタジオ経済圏」を構築する**ことで、中長期的な収益化を図る。

### 収益源の3本柱（＋α）

| 収益源 | 期待度 | 内容・開始時期 |
|:---|:---|:---|
| **1. 自社ポータルのリワード広告** | ★★★ | 「動画を見てコンティニュー」「資金2倍」などの高単価なWeb動画広告。YouTube経由の流入が増えるほど即座に収益に直結する。（即時可能） |
| **2. スタジオサポーター（Stripe）** | ★★★ | ファン向けの月額課金制度（例: 月額300円で全ゲーム広告非表示＋限定バッジ等）。Tetra LMSのStripe基盤を流用可能。（Phase 2以降） |
| **3. YouTube AdSense** | ★★☆ | コンテンツ再生による広告収益（登録者1,000人達成後） |
| **技術ブランディング** | ★★★ | （※副次効果）法人のB2B案件獲得や採用への強固なポートフォリオとして機能 |

### 月間収益シミュレーション（AdSense）

技術系チャンネルのCPM目安: ¥500〜1,500（日本）

| 時期 | 登録者 | 月間再生 | 月間広告収入 |
|:---|:---|:---|:---|
| 3ヶ月目 | 100〜300 | 2,000〜5,000 | ¥0（条件未達） |
| 12ヶ月目 | 1,000〜3,000 | 20,000〜50,000 | ¥1〜5万 |
| 24ヶ月目 | 5,000〜10,000 | 100,000〜200,000 | ¥5〜20万 |

### コスト

| 項目 | 月額 |
|:---|:---|
| Gemini API | ¥0〜3,000 |
| Vercel | ¥0（Hobby Plan） |
| ドメイン | 〜¥200 |
| **合計** | **¥0〜3,000/月** |

### 結論

| 観点 | 評価 |
|:---|:---|
| YouTube AdSense一本で稼ぐ | ❌ 難易度が高く不安定 |
| エコシステム（YouTube→自社ゲーム） | ✅ **熱量の高いファンを自社経済圏（広告・サブスク）へ誘導できる最強のループ** |
| コスト回収 | ✅ 運用コストがほぼゼロなので即座に黒字化可能 |
| 楽しさ対効果 | ✅ 趣味として最高効率 |

> **真の価値は「技術ブランド」と「独自のスタジオプラットフォームの構築」**にある。YouTubeは集客装置であり、真の収益基盤はゲームポータル側に育っていく。
