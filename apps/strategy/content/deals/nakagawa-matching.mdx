## 1. 案件概要

### 1.1. 背景

中川圭介氏が運営する業界特化型マッチングサイト（ビザスクの業界特化版）のリニューアル。既存サイトはパッケージ型SaaSで構築されており、以下の課題がある:

- ターゲットのミスマッチ（現場の課題解決 → 実際のニーズはマーケ・市場調査）
- パッケージ型の柔軟性不足（動線・文言の変更ができない）
- サービスが伝わらないUI
- 登録〜相談の動線が複雑

### 1.2. 方向性の転換

- **ターゲット:** 現場の課題解決 → **新規事業開発・マーケティング担当者向けの市場調査**
- **回答者側:** 一定の集客基盤あり → **相互マッチング機能を追加**
- **決済機能:** 初期は不要（マッチングの活発化が優先）

> マッチングサイトのリニューアルは「既存SaaSの作り直し」であり、仕様の正解がある。展開力・想像力より「決まったものを管理する」力が求められるため、山田さんの適性に合う可能性がある。

## 9. 提案書（クライアント向け）

> 以下は4/20 MTGで中川様に提示する提案内容。代替案（Phase A ディスカウント）ベース。

### 9.1. 現行サイトの課題

| 課題 | 詳細 |
|:---|:---|
| **ターゲットのミスマッチ** | 現場の課題解決ニーズは即時性が高く、サイトの利用に至らなかった |
| **パッケージ型の柔軟性不足** | 動線の変更・文言修正など、細かなチューニングができない |
| **サービスが伝わらないUI** | ファーストビューで「何ができるサイトか」が直感的に理解できない |
| **登録〜相談の動線が複雑** | 登録までのステップが多く、登録後のやり取りも煩雑 |
| **回答者の相互マッチング未対応** | 回答者がスキル・対応分野を登録してマッチングする仕組みがない |

### 9.2. ご提案するアプローチ: AI伴走型開発

| | 従来型受託開発 | AI伴走型開発 |
|:---|:---|:---|
| **設計** | 要件定義に2〜3ヶ月 → 仕様確定 → 開発 | 要件定義と並行してプロトタイプを構築。**動くものを見ながら仕様を固める** |
| **開発** | エンジニアチーム（5〜10名）で数ヶ月 | **最新のAI技術を活用し、** 少数精鋭チームが設計・品質管理に集中 |
| **確認** | 開発完了後に全体像を確認 | **週1回のプロトタイプ確認。** 毎週画面を見てフィードバック |
| **方向修正** | 仕様変更 → 追加見積もり → 再開発 | **月次単位で柔軟に調整。** 方向修正のコストが低い |

**開発サイクル:**

```
① ご要望をヒアリング → ② 翌週にプロトタイプを提示 → ③ フィードバック → ④ 即反映 → ①に戻る
```
