## 2. Phase 1: sGTMブリッジサーバー

### 2.1. 背景と課題

LINE友だち追加はLINEアプリ内で完結するため、**クライアントサイドGTMでは計測不可能**。
広告クリック（Google Ads / META / LINE広告）から友だち追加までのCV計測には、サーバーサイドでの突合が必要。

| 課題 | 詳細 |
|:---|:---|
| **Cookie断絶** | 広告LP（Webブラウザ）→ LINEアプリ間でCookieが引き継がれない |
| **計測不能** | 友だち追加完了はLINE内で発生し、GTMタグが発火しない |
| **広告最適化への影響** | CVデータが広告媒体に返らないため、機械学習の最適化が効かない |

### 2.2. アーキテクチャ

```
広告クリック → LP（gclid/fbclid/ldtag保存）→ LIFF（友だち追加 + クリックID送信）
                                                    │
                                LINE Webhook（follow）│
                                                    ▼
                                          ┌──────────────────┐
                                          │ Bridge Server    │
                                          │ - Webhook受信     │
                                          │ - 署名検証        │
                                          │ - クリックID突合   │
                                          └────────┬─────────┘
                                                   │
                                                   ▼
                                          ┌──────────────────┐
                                          │ sGTM Container   │
                                          └────────┬─────────┘
                                                   │
                                        ┌──────────┼──────────┐
                                        ▼          ▼          ▼
                                   Google Ads   GA4      LINE広告 CAPI
```

### 2.3. 主要コンポーネント

#### A. Webhook受信エンドポイント

LINE Messaging API の follow/unfollow イベントを受信。

| 項目 | 仕様 |
|:---|:---|
| **エンドポイント** | `POST /api/webhook/line` |
| **署名検証** | `X-Line-Signature` ヘッダーをChannel Secretで検証（HMAC-SHA256） |
| **対象イベント** | `follow`（友だち追加）、`unfollow`（ブロック） |
| **レスポンス** | 200 OK を即座に返却（非同期処理） |

> ⚠️ **セキュリティ要件:** 署名検証は必須。検証なしでWebhookを受け付けてはならない。

#### B. クリックID保存・突合

LIFF経由で取得した広告クリックIDとLINE userIDを紐付ける。

| 項目 | 仕様 |
|:---|:---|
| **保存対象** | `gclid`（Google Ads）、`fbclid`（META）、`ldtag`（LINE広告） |
| **突合キー** | LINE `userId` |
| **保存先** | KVストア（TTL付き） |
| **TTL** | 90日（gclid有効期限に合わせる） |
| **データフロー** | LP → LocalStorage保存 → LIFF起動時にクエリパラメータとして渡す → サーバーに送信 |

```
// LIFFからサーバーへの送信ペイロード
{
  "lineUserId": "U1234567890abcdef",
  "gclid": "EAIaIQobChMI...",
  "fbclid": "IwAR3...",
  "ldtag": "...",
  "liffRedirectUri": "https://liff.line.me/{LIFF_ID}",
  "timestamp": "2026-05-03T17:00:00+09:00"
}
```

#### C. sGTMイベント送信

Measurement Protocol（GA4形式）で sGTM コンテナにイベントを転送。

| 項目 | 仕様 |
|:---|:---|
| **送信先** | `POST https://sgtm.{domain}/mp/collect` |
| **プロトコル** | GA4 Measurement Protocol |
| **イベント名** | `line_follow`（友だち追加）、`line_unfollow`（ブロック） |
| **パラメータ** | `gclid`, `client_id`, `user_id`（LINE userId）, `timestamp_micros` |

```
// sGTMへの送信ペイロード（Measurement Protocol GA4形式）
{
  "client_id": "LINE_U1234567890abcdef",
  "user_id": "U1234567890abcdef",
  "timestamp_micros": 1778094000000000,
  "events": [{
    "name": "line_follow",
    "params": {
      "gclid": "EAIaIQobChMI...",
      "source": "line",
      "medium": "social",
      "campaign": "futara_games_launch"
    }
  }]
}
```

#### D. LIFFランディングページ

友だち追加導線 + クリックIDをサーバーに送信するフロントエンド。

| 項目 | 仕様 |
|:---|:---|
| **LIFF SDK** | `@line/liff` v2.x |
| **処理フロー** | ① LIFF初期化 → ② URLからクリックID取得 → ③ サーバーに送信 → ④ 友だち追加画面へリダイレクト |
| **フォールバック** | LIFF外アクセス時はWebブラウザでLP表示 + QRコード表示 |

### 2.4. sGTMコンテナ側の設定

| 設定項目 | 内容 |
|:---|:---|
| **Client** | Measurement Protocol (GA4) Client — `line_follow` イベントをクレーム |
| **Trigger** | Custom Event: `event_name == "line_follow"` |
| **Tags** | ① GA4 Tag（GA4プロパティに送信）② Google Ads Conversion Tag（gclid突合でCV計上）③ LINE広告 CAPI Tag（Conversion API送信） |

> **Enhanced Conversions:** ハッシュ化したメールアドレス・電話番号を送信することで、gclid不一致時のフォールバック突合が可能。LINE Profile APIからメールアドレスを取得できる場合に有効。

### 2.5. LINE広告 CAPI（Conversion API）連携

2026年4月にLINE広告とYahoo!ディスプレイ広告が統合され「LINEヤフー広告」に。CAPI仕様もアップデート。

| 項目 | 仕様 |
|:---|:---|
| **API** | LINEヤフー広告 Conversion API |
| **認証** | ビジネスマネージャーのアクセストークン |
| **送信データ** | イベントタイプ、タイムスタンプ、ハッシュ化PII（email/phone） |
| **重複排除** | `event_id` + `ldtag` でデデュプリケーション |
| **併用方針** | LINE Tag（クライアントサイド）と CAPI（サーバーサイド）を併用し、計測精度を最大化 |

## 3. Phase 2: 配信エンジン

### 3.1. 機能要件

L-STEP等の外部ツールを使わず、LINE Messaging APIで直接配信を制御する。

| 機能 | 概要 | 優先度 |
|:---|:---|:---|
| **セグメント配信** | タグ・属性・行動履歴に基づくターゲティング配信 | 🔴 高 |
| **ステップ配信** | 友だち追加後の自動シナリオ配信（Day 0, 1, 3, 7...） | 🔴 高 |
| **リッチメニュー切替** | ユーザー属性に応じたリッチメニューの動的切替 | 🟡 中 |
| **一斉配信** | 全友だちへの配信（管理画面の代替） | 🟡 中 |
| **1:1チャット** | 手動応対（管理画面併用で可） | 🟢 低 |

### 3.2. データモデル（ユーザーDB）

```sql
-- 友だち情報
CREATE TABLE line_users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  line_user_id TEXT UNIQUE NOT NULL,
  display_name TEXT,
  picture_url TEXT,
  status TEXT DEFAULT 'active', -- active / blocked
  followed_at TIMESTAMPTZ NOT NULL,
  blocked_at TIMESTAMPTZ,
  -- 広告突合
  gclid TEXT,
  fbclid TEXT,
  ldtag TEXT,
  -- 属性
  tags JSONB DEFAULT '[]',
  custom_fields JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- 配信シナリオ
CREATE TABLE step_scenarios (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id UUID NOT NULL, -- LINE公式アカウント
  name TEXT NOT NULL,
  trigger_event TEXT NOT NULL, -- 'follow', 'tag_added', 'custom'
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- ステップ配信メッセージ
CREATE TABLE step_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scenario_id UUID REFERENCES step_scenarios(id),
  delay_minutes INTEGER NOT NULL, -- トリガーからの遅延（分）
  message_type TEXT NOT NULL, -- 'text', 'flex', 'image', 'video'
  message_payload JSONB NOT NULL, -- Messaging API のメッセージオブジェクト
  sort_order INTEGER NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 配信ログ
CREATE TABLE delivery_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  line_user_id TEXT NOT NULL,
  scenario_id UUID,
  message_id UUID,
  message_type TEXT NOT NULL, -- 'step', 'segment', 'broadcast'
  status TEXT NOT NULL, -- 'sent', 'failed', 'scheduled'
  sent_at TIMESTAMPTZ,
  error_message TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### 3.3. ステップ配信の実行フロー

```
1. follow Webhook受信
2. line_users に INSERT
3. step_scenarios からトリガー = 'follow' のシナリオを取得
4. step_messages を delay_minutes 順に取得
5. 各メッセージを delay 後にキューに投入
6. キューワーカーが Messaging API Push Message を実行
7. delivery_logs に結果を記録
```

> **キュー実装:** Vercel Cron + KV / Upstash QStash / Cloud Tasks のいずれか。
> 遅延実行が必要なため、単純なHTTPリクエストではなくキューベースの設計が必須。

### 3.4. LLM活用（SOCT内部のみ）

| 活用箇所 | 内容 | クライアントへの見せ方 |
|:---|:---|:---|
| **メッセージ生成** | 配信テンプレートのパーソナライズ生成 | 「独自の運用ノウハウ」 |
| **セグメント提案** | ユーザー行動データから最適セグメントを提案 | 「データに基づく分析」 |
| **配信タイミング最適化** | 開封率データからの最適送信時刻推定 | 「独自の分析手法」 |
| **レポート自動生成** | 月次レポートのナラティブ生成 | 「専門コンサルタントによる分析」 |

## 4. Phase 3: 分析ダッシュボード

### 4.1. 機能要件

| 機能 | 概要 | データソース |
|:---|:---|:---|
| **友だち数推移** | 日次/月次の友だち増減グラフ | line_users テーブル |
| **ファネル分析** | 広告クリック → LP → LIFF → 友だち追加 の転換率 | KVストア + line_users |
| **配信効果** | 配信ごとの開封率・クリック率・ブロック率 | Messaging API Insight + delivery_logs |
| **セグメント分析** | タグ別・属性別の行動傾向 | line_users + delivery_logs |
| **月次レポート** | PDF/HTML形式の自動生成レポート | 全テーブル + LLM |
| **広告ROAS** | gclid/fbclid別のCV数・CPA算出 | line_users（広告突合データ） |

### 4.2. レポート自動生成（LLM連携）

```
1. 月末にCronで集計SQLを実行
2. 集計結果をLLM（Claude/GPT）に投入
3. ナラティブ形式のレポートを生成
4. PDF/HTMLに変換
5. クライアントにメール or Slack通知
```

> **golf-performance-sns.mdx §3.3 との整合:** 「分析レポート → SOCT独自 → 月次で共有」の方針をそのまま実装。

### 5.4. マルチテナント設計

全アプリ共通で、クライアント（広告アカウント）単位の管理を行う。

```sql
-- 共通: クライアント管理（packages/db）
CREATE TABLE clients (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL, -- 「Futara Games」「ゴルフパフォーマンス」等
  created_at TIMESTAMPTZ DEFAULT now()
);

-- apps/line 固有
CREATE TABLE line_accounts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  client_id UUID REFERENCES clients(id),
  channel_id TEXT UNIQUE NOT NULL,
  channel_secret TEXT NOT NULL, -- 暗号化して保存
  channel_access_token TEXT NOT NULL, -- 暗号化して保存
  sgtm_endpoint TEXT,
  liff_id TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- apps/meta 固有（将来）
CREATE TABLE meta_ad_accounts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  client_id UUID REFERENCES clients(id),
  pixel_id TEXT NOT NULL,
  access_token TEXT NOT NULL, -- 暗号化して保存
  created_at TIMESTAMPTZ DEFAULT now()
);
```

> `client_id` で全テーブルを横断的に紐付け。RLS でクライアント単位のアクセス制御を行う。

|:---|:---|:---|

| **月次レポート自動生成** | 全KPIを集計し、LLMがナラティブ形式のレポートをPDF/HTML出力 | 「専門コンサルタントの分析レポート」 |
| **自然言語チャットボット** | キーワード一致ではなくLLMで文脈を理解した自動応答 | 「高品質な自動応答」 |
| **広告ROAS統合分析** | gclid/fbclid突合で「広告費→友だち追加→CV」の一気通貫ROAS | 「広告とLINEを直結した分析」 |
| **A/Bテスト自動化** | メッセージ・配信時刻のバリアント自動配分→統計判定→勝者自動採用 | 「科学的な効果検証」 |

> **結論:** L-STEPプロプラン（¥32,780/月）相当の全機能をカバーしつつ、LLM活用による7つの独自機能で**価格は≒¥0、機能は上位互換**を実現する。複数クライアントに展開すれば、L-STEPライセンス費の削減分（1社¥32,780/月 × N社）がそのまま粗利になる。
